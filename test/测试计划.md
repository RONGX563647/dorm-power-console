## 3. 测试环境（与仓库示例对齐）

推荐使用基于容器的测试环境（docker-compose），示例服务：

- `postgres`：数据库
- `mqtt`：EMQX 或 Mosquitto（按需求）
- `backend-java`：运行 Java 后端（端口 8000）
- `backend-py`：运行 FastAPI 后端（端口 8000 或不同端口，避免冲突）
- `frontend`：Next.js（开发模式或构建后静态服务）

示例 `docker-compose`（简化，基于仓库示例与 `.env.example`）：

```yaml
version: "3.8"
services:
  postgres:
    image: postgres:13
    environment:
      POSTGRES_USER: dorm_user
      POSTGRES_PASSWORD: change_me
      POSTGRES_DB: dorm_power
    volumes:
      - pgdata:/var/lib/postgresql/data

  mqtt:
    image: emqx/emqx:4.4.3
    ports:
      - "1883:1883"
      - "18083:18083"

  backend-java:
    build: ./backend
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/dorm_power
      SPRING_DATASOURCE_USERNAME: rongx
      SPRING_DATASOURCE_PASSWORD: "<set_secure_password>"
    ports:
      - "8000:8000"
    depends_on:
      - postgres
      - mqtt

  backend-py:
    build: ./back_end
    environment:
      DATABASE_URL: postgresql+psycopg://dorm_user:change_me@postgres:5432/dorm_power
      MQTT_ENABLED: "1"
      MQTT_HOST: mqtt
    ports:
      - "8001:8000"
    depends_on:
      - postgres
      - mqtt

  frontend:
    build: ./dorm-power-console
    environment:
      BACKEND_BASE_URL: http://localhost:8000
    ports:
      - "3000:3000"

volumes:
  pgdata:
```

注意：上面为示例，实际端口/凭据请在 CI/CD 环境中通过安全方式注入（Secrets/ENV）。

## 5. 详细测试项（补充规范化测试用例说明）

每个功能模块至少包含下面几类测试用例：

- 正常流（Happy path）
- 边界条件（参数最小/最大/空）
- 异常流（非法输入、设备离线、网络中断）
- 并发与性能（并发命令、并发连接）
- 安全（认证失败、越权、注入攻击）

下面给出若干规范化示例用例：

用例ID: TC-DEVICE-REG-001
标题: 设备自动注册（正常流）
前置条件:

- 后端服务可用（Java 或 Python 后端任一）
- MQTT Broker 可达
  测试步骤:

1.  模拟设备通过 MQTT 发布首次上线消息（topic: dorm/status）
2.  查询设备列表接口 `/api/devices` 或 WebSocket 订阅更新
    预期结果:

- 后端创建设备记录并返回 201（或在设备列表可见）
- 设备状态为 online
  附加检查:
- 数据库中存在该设备的记录（核对 device_id、room 等字段）

用例ID: TC-CMD-001
标题: 下发开关命令并验证 ACK
前置条件:

- 设备已注册并在线
  测试步骤:

1.  调用下发命令接口 `/api/command` 下发开（on）命令
2.  观察 MQTT 消息下发到设备 topic
3.  模拟设备返回 ACK
    预期结果:

- 命令状态变为 SUCCESS
- 数据库记录命令执行时间与结果

用例ID: TC-PERF-001
标题: 并发 1000 台虚拟设备连接压力测试
环境:

- 使用虚拟设备模拟器（参考 `back_end/tools/simulate_device.py`）
  步骤与期望:
- 逐步启动模拟器到 1000 连接，HTTP/API 响应在 95% 请求中 < 500ms
- 监控内存/CPU 与数据库连接数，确保无明显资源泄露

## 5.A 功能需求（Functional Requirements）

以下功能需求源自需求文档与代码库，测试时需逐条验证：

- FR-01 设备自动注册：设备首次上线时能自动注册并记录设备基本信息（device_id、room、model）。
- FR-02 设备管理：支持设备查询、分页、筛选、信息修改与删除（权限受限）。
- FR-03 远程控制：支持向设备下发开/关、模式切换、定时命令，并能查询命令执行状态与历史记录。
- FR-04 遥测采集：按采样频率接收并存储设备遥测数据（功率、电压、电流、能耗等）。
- FR-05 实时通信：使用 MQTT 与 WebSocket 实现设备与前端的实时消息推送与订阅。
- FR-06 用户认证与授权：支持用户登录、JWT 鉴权、角色权限控制（管理员、宿管、学生）。
- FR-07 报表与AI分析：生成历史用电报表与 AI 分析结果（异常检测、节能建议）。
- FR-08 健康检查与监控：提供 /health 或相应接口用于监控（HTTP 状态、MQTT 连接数、DB 可用性）。
- FR-09 审计日志：关键操作（登录、命令下发、权限变更）应写入审计日志，便于追溯。
- FR-10 批量操作与定时任务：支持对设备批量下发命令与定时策略管理。
- FR-11 数据导出与备份：支持 CSV/JSON 导出和定期数据库备份（测试验证备份可用性）。
- FR-12 错误与异常处理：在设备离线、命令超时、网络异常时给出明确错误码与处理逻辑。

每项功能需求应至少对应一个或多个测试用例（单元/集成/系统/验收）。

## 5.B 非功能需求（Non-Functional Requirements）

下列非功能需求应以可量化指标形式进入测试和验收：

- NFR-Performance-1（响应时间）: API 响应 P95 ≤ 500ms（在正常负载下），P99 ≤ 1s（参考负载）。
- NFR-Performance-2（并发能力）: 支持至少 1000 个并发 MQTT 设备连接且消息丢失率 ≤ 1%。
- NFR-Performance-3（吞吐量）: 后端能处理至少 5000 条/分钟的遥测写入，且 DB 写入成功率 ≥ 99.9%。
- NFR-Availability（可用性）: 系统月度可用性 ≥ 99.5%。
- NFR-Reliability（可靠性）: 数据持久性 RPO ≤ 0（事务提交即持久），关键数据丢失率 ≤ 0.1%。
- NFR-Security（安全）: 无高危/严重漏洞（安全扫描），所有 API 强制 HTTPS/WSS，JWT 密钥长度与过期策略满足安全要求。
- NFR-Scalability（可扩展）: 系统能水平扩展后端实例，并在扩容后线性提升吞吐。
- NFR-Maintainability（可维护）: 代码自动化测试覆盖率 ≥ 80%（单位+集成），CI 检查通过率 100%。
- NFR-Compatibility（兼容性）: 前端支持主流浏览器（Chrome/Edge/Firefox）近两个主版本；设备协议兼容既有设备。
- NFR-Privacy（隐私）: 敏感数据（密码/令牌）在传输层与存储层均加密；日志脱敏策略执行。

## 5.C 测试标准与验收准则（Acceptance Criteria / 测试标准）

下面为发布/验收的量化测试标准，所有条件满足则可视为通过：

1. 功能覆盖与正确性

- 功能测试用例执行率 ≥ 98%，通过率 ≥ 98%（仅计非阻塞性故障）。
- 所有 FR-01 至 FR-12 的关键用例通过，且关键路径（设备注册→下发命令→ACK→遥测入库）全链路验证通过。

2. 回归与自动化

- 单元测试 + 集成测试自动化套件在 CI 上 Green（通过），代码覆盖率 ≥ 80%。

3. 性能与稳定性

- API 响应符合 NFR-Performance-1 的指标（P95 ≤ 500ms）。
- 并发 1000 MQTT 连接测试中，消息丢失率 ≤ 1%，系统无崩溃或内存泄露迹象。
- 性能测试中 DB 写入成功率 ≥ 99.9%，并发下错误率 ≤ 1%。

4. 可用性与恢复

- 系统在连续 7 天稳定运行中可用性≥99.5%。
- 恢复测试（断电/进程重启）后关键服务恢复时间（RTO）≤ 10 分钟，最近备份可用于恢复且 RPO ≤ 1 小时（针对测试环境验证）。

5. 安全与合规

- 安全扫描（OWASP ZAP / SAST）无 Critical/High 漏洞；发现的 Medium/Low 漏洞须有处理计划。
- 所有对外接口强制使用 TLS 1.2+；JWT Secret 密钥长度与过期策略满足安全团队要求。

6. 缺陷准则

- 发布前 P0、P1 缺陷必须全部关闭；P2 缺陷修复或有合理延期计划并获项目经理批准。
- P0 缺陷响应时间 ≤ 24 小时，P1 缺陷响应时间 ≤ 3 个工作日。

7. 日志与监控

- 关键指标（CPU、内存、MQTT 连接数、DB 连接数、API 错误率）监控就绪并报警策略配置完毕。

8. 兼容性与可用性验收

- 前端在 Chrome/Edge/Firefox 最新与次新版本功能无阻塞性问题。

若任一关键测试标准未满足，应记录在测试总结并触发回归或阻塞发布。对于可接受的轻微偏差（如性能波动），需在变更请求中明确缓解措施与时间表。

# 第五章：监控告警与系统运维设计

> 前四章解决了“系统怎么搭、请求怎么进、业务怎么跑、设备怎么实时联动”的问题。本章收束到一个成熟后端系统不可回避的最后一环：系统如何监控自己、如何发现异常、如何通知人、如何留痕、如何备份和清理。

---

## 教学目标与安排

### 学习目标

1. 能说明监控、告警、通知、日志、备份在当前项目中的职责分工。
2. 能读懂系统指标采集、设备告警生成、通知流转和日志沉淀的主要调用链。
3. 能理解“监控发现问题”和“运维处置问题”之间的区别与衔接。
4. 能识别当前实现中的工程优点与潜在风险。
5. 能把后端系统理解为“业务能力 + 运维能力”的整体，而不是只关注接口开发。

### 建议课堂安排

- 导入问题：一个系统如果能跑业务，但出了问题没人知道、没人定位、也没人恢复，算不算完成？
- 第一部分：监控指标如何采集与查询
- 第二部分：告警如何从阈值判断走到实时推送
- 第三部分：通知、日志、备份如何形成处置闭环
- 实践任务：画出一次异常功率告警从产生到通知的链路

## 1. 运维能力总体架构

当前项目中的运维能力大致可以抽象为下面这条链：

```text
系统运行 / 设备上报 / 用户操作
  ↓
MonitoringService 采集指标
AlertService 判断是否越界并生成告警
  ↓
WebSocketNotificationService / NotificationService 对外通知
  ↓
SystemLogService 记录过程与结果
  ↓
DataBackupService + Scheduler 提供备份和清理
```

如果从“发现问题到处理问题”的角度看，可以再简化成：

```text
监控 → 告警 → 通知 → 日志 → 备份/清理
```

教学时要让学生建立一个完整认知：

- 监控负责“看见状态”。
- 告警负责“识别异常”。
- 通知负责“让人知道”。
- 日志负责“留下证据”。
- 备份负责“在出事后还能恢复”。

## 2. 监控指标采集与查询

### 2.1 核心服务：MonitoringService

`MonitoringService` 是当前项目监控能力的核心服务，它主要做三件事：

1. 收集系统运行指标。
2. 汇总设备在线状态。
3. 查询历史指标和 API 性能统计。

### 2.2 指标采集内容

`collectSystemMetrics()` 当前采集的重点包括：

- CPU 负载
- 堆内存使用率与已用内存
- JVM 运行时长

典型实现如下：

```java
double cpuUsage = osMXBean.getSystemLoadAverage();
long usedMemory = memoryMXBean.getHeapMemoryUsage().getUsed();
long maxMemory = memoryMXBean.getHeapMemoryUsage().getMax();
double memoryUsagePercent = maxMemory > 0 ? (double) usedMemory / maxMemory * 100 : 0;
saveMetric("SYSTEM", "cpu_usage", cpuUsage, "%");
saveMetric("SYSTEM", "memory_usage", memoryUsagePercent, "%");
saveMetric("SYSTEM", "uptime_minutes", uptime / 60000.0, "minutes");
```

这说明当前项目并没有依赖外部监控平台才能进行基础监控，而是先把关键指标直接落到自己的 `SystemMetrics` 数据里。

### 2.3 设备状态监控

`getDeviceStatus()` 则从业务角度统计设备在线情况：

- 总设备数
- 在线设备数
- 离线设备数
- 在线率

它不是直接信任 `device.isOnline()`，而是同时结合 `lastSeenTs` 做超时判断，这体现了一个实际工程意识：**设备状态要结合时间窗口判断，不能只看某个布尔字段。**

### 2.4 管理接口：MonitoringController

监控能力并不是只在后台静默运行，项目还暴露了管理接口：

- `/api/admin/monitor/system` 获取系统状态
- `/api/admin/monitor/devices` 获取设备在线统计
- `/api/admin/monitor/api-performance` 获取 API 性能统计
- `/api/admin/monitor/metrics` 获取历史指标
- `/api/admin/monitor/collect` 手动触发采集
- `/api/admin/monitor/cleanup` 清理过期指标

这很适合拿来讲“监控服务”和“监控接口”的分工：

- Service 负责采集与统计。
- Controller 负责把这些能力开放给管理端页面或运维人员。

### 2.5 定时采集配合

第四章已经看到 `SystemMetricsScheduler` 会周期性调用：

```java
monitoringService.collectSystemMetrics();
```

这说明监控数据的来源有两条：

1. 系统按计划自动采集。
2. 管理员也可以手动触发采集。

这种设计既适合日常运维，也方便课堂演示。

## 3. 告警生成与通知链路

### 3.1 核心服务：AlertService

`AlertService` 负责设备告警的生成、查询、解决和配置管理。它的核心逻辑是：

1. 读取设备告警阈值配置。
2. 判断当前数值是否越界。
3. 生成 `DeviceAlert` 并落库。
4. 通过 WebSocket 向前端推送告警。

### 3.2 典型逻辑：checkAndGenerateAlert()

这个方法当前处理 `power`、`voltage`、`current` 三类告警。核心判断模式是：

```java
if (value > config.getThresholdMax()) {
    level = "warning";
    message = "Power exceeds threshold...";
} else if (value < config.getThresholdMin()) {
    level = "info";
    message = "Power below threshold...";
}
```

一旦命中阈值，它会：

- 生成告警实体并保存。
- 输出日志。
- 调用 `webSocketNotificationService.broadcastAlert()` 实时推送。

所以当前项目的告警链路不是“只落库不通知”，而是已经具备实时反馈能力。

### 3.3 告警是如何被触发的

当前最典型的触发点在 `MqttBridge.handleTelemetryMessage()`：

```text
设备上报功率/电压/电流
  ↓
MqttBridge 接收 telemetry
  ↓
AlertService.checkAndGenerateAlert(deviceId, type, value)
  ↓
DeviceAlertRepository.save(alert)
  ↓
WebSocketNotificationService.broadcastAlert(...)
```

这条链非常适合课堂演示，因为它把“设备数据流”和“告警流”自然串联起来了。

### 3.4 告警管理接口：AlertController

项目中已经提供了完整的告警管理 REST 接口，包括：

- 查询指定设备告警
- 查询所有未解决告警
- 解决告警
- 查询与更新告警阈值配置
- 模拟告警

其中 `simulateAlarm()` 是一个很适合教学和联调的接口，因为它让学生不必等待真实设备越界，也能立即验证整条告警链路。

### 3.5 通知中心：NotificationService 与 NotificationController

除了实时 WebSocket 告警，项目还提供了通知中心能力，用来沉淀和管理用户通知。`NotificationService` 负责：

- 创建系统通知
- 创建告警通知
- 创建邮件通知
- 查询通知列表与未读数
- 管理通知偏好
- 支持免打扰时段

对应的 `NotificationController` 暴露了通知列表、未读数量、统计、测试邮件和偏好设置接口。

这里很适合教学上强调一件事：

- WebSocket 通知偏实时、偏瞬时。
- Notification 通知偏持久、偏中心化管理。

这两者并不是重复建设，而是服务不同的交互需求。

## 4. 系统日志与审计留痕

### 4.1 核心服务：SystemLogService

`SystemLogService` 提供了较完整的日志记录与查询能力，支持：

- 信息日志 `info()`
- 警告日志 `warn()`
- 错误日志 `error()`
- 审计日志 `audit()`

这使得系统既能记录运行状态，也能记录用户操作和安全敏感行为。

### 4.2 日志的几种来源

当前项目里的日志来源主要有三类：

1. **系统运行日志**：如备份成功、清理失败、设备上下线。
2. **错误日志**：如任务执行异常、备份失败。
3. **审计日志**：通过 `@AuditLog` 标记的管理操作。

例如：

- `AlertController.resolveAlert()` 带有 `@AuditLog`
- `DataBackupController.createDatabaseBackup()` 带有 `@AuditLog`

这说明当前项目已经把“管理动作可追踪”纳入设计范围。

### 4.3 管理接口：SystemLogController

日志管理接口当前支持：

- 分页查看全部日志
- 按级别查看
- 按类型查看
- 按用户名查看
- 关键词搜索
- 查看统计
- 清理过期日志

这很适合告诉学生：日志不是只写文件，还要能被结构化查询和管理。

### 4.4 课堂讲解重点

在这一节里，要特别强调“日志”和“告警”的差别：

- 日志关注的是记录事实。
- 告警关注的是提醒异常。

两者经常一起出现，但不能混为一谈。

## 5. 数据备份与清理机制

### 5.1 核心服务：DataBackupService

`DataBackupService` 负责两类备份：

1. 数据库备份 `createDatabaseBackup()`
2. 数据导出备份 `createDataExportBackup()`

其中数据库备份当前通过 `pg_dump` 执行，导出备份则将数据打包为 zip。

### 5.2 备份流程

以数据库备份为例，当前流程是：

```text
创建 DataBackup 记录
  ↓
调用 pg_dump 执行备份
  ↓
根据结果更新状态、文件大小和完成时间
```

这说明备份在当前项目中不是单纯执行一个系统命令，而是被建模成了可以查询、统计、删除的业务对象。

### 5.3 备份管理接口：DataBackupController

项目中已提供：

- 查看全部备份
- 查看最近备份
- 创建数据库备份
- 创建导出备份
- 删除备份
- 清理过期备份
- 查看备份统计

同时，创建和删除备份等操作带有 `@AuditLog`，这表明备份能力已经进入管理审计范围。

### 5.4 清理机制

备份和日志并不是越多越好，项目已经实现了基于保留天数的清理逻辑：

- `cleanupOldBackups(retentionDays)`
- `cleanupOldLogs(retentionDays)`
- `cleanupOldMetrics(retentionDays)`

再结合 `SystemMetricsScheduler` 的定时清理，就形成了一个基本的运维闭环：

```text
采集数据
  ↓
长期积累
  ↓
按保留策略清理
```

这能帮助学生理解：运维设计不仅是“多收集”，还包括“有节制地保留和清理”。

## 6. 当前实现特点与工程提醒

### 6.1 当前实现的优点

从教学角度看，当前项目的运维能力有几个明显优点：

1. 监控、告警、通知、日志、备份都有独立服务，职责比较清楚。
2. 实时告警已经打通到 WebSocket 推送，不是只做离线存储。
3. 管理接口较完整，适合教学演示和前后端联调。
4. 备份、清理和统计都被纳入统一服务，不是零散脚本拼接。
5. `@AuditLog` 已用于部分关键管理操作，具备基础审计意识。

### 6.2 当前实现中的工程提醒

教学时也要提醒学生，这套实现仍有一些值得继续优化的地方：

1. `MonitoringService` 当前主要依赖 JVM 和本机资源指标，适合作为基础实现，但对分布式场景支持有限。
2. `DataBackupService` 中 `pg_dump` 的主机、用户名和密码仍是较固定的实现方式，生产环境应进一步配置化并注意凭据安全。
3. 告警通知当前更偏 WebSocket 和系统通知，若面向真实生产，通常还要接入短信、邮件、企业 IM 等外部通道。
4. 通知接口使用 `username` 作为请求参数的模式，在更严格系统中通常还应结合当前登录态做约束。
5. 定时清理和备份虽然有实现，但调度启用点仍应像上一章那样继续核实，不能只看 `@Scheduled` 注解就默认运行生效。

### 6.3 推荐演进方向

可以引导学生思考下面这些高级优化方向：

1. 将监控指标进一步接入 Actuator、Prometheus 或 Grafana 体系。
2. 为告警增加去重、抑制和升级策略，避免告警风暴。
3. 将通知中心和用户权限、角色、值班策略进一步关联。
4. 让备份任务支持异步执行、状态跟踪和失败重试。
5. 为日志与监控增加统一检索和可视化看板。

## 7. 本章小结

### 关键知识点

1. `MonitoringService` 负责收集和查询系统与设备指标。
2. `AlertService` 负责基于阈值生成告警，并通过 WebSocket 实时推送。
3. `NotificationService` 负责持久化通知中心与用户通知偏好。
4. `SystemLogService` 负责系统日志、错误日志和审计日志沉淀。
5. `DataBackupService` 负责备份、清理和备份统计，是运维恢复能力的重要组成部分。

### 实践建议

1. 让学生画出一次异常电流值从 `MqttBridge` 进入，到 `AlertService` 生成告警，再到 WebSocket 推送和通知沉淀的流程图。
2. 让学生比较监控、告警、通知、日志四者的职责边界。
3. 让学生检查哪些管理接口已经加了 `@AuditLog`，哪些还没有。
4. 让学生思考如果备份失败，当前系统如何发现、记录和补救。

### 教学收束建议

至此，这套后端教学材料已经覆盖了：

- 项目架构与技术栈
- 安全、中间件与异常治理
- 业务逻辑与数据层设计
- 设备通信与实时交互
- 监控、告警与系统运维

如果继续扩展教学，可以进入综合实战，例如：新增一个设备管理功能，要求同时补齐接口、安全校验、实时推送、日志与监控埋点。

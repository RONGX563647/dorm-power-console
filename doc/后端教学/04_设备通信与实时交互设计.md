# 第四章：设备通信与实时交互设计

> 从这一章开始，教学重点从传统的增删改查后端，进入“设备系统后端”的核心能力：如何接设备消息，如何做状态同步，如何把后端变化实时推到前端。当前项目里，这条链路主要由 MQTT、WebSocket 和定时任务共同完成。

## 1. 实时通信总体架构

在当前项目中，设备通信链路可以概括为下面这条主线：

```text
设备端
  ↓ MQTT 发布状态/遥测/应答/事件
MQTT Broker
  ↓
MqttBridge 订阅消息并解析
  ├─ 更新设备状态
  ├─ 写入遥测数据
  ├─ 处理命令应答
  ├─ 触发告警
  └─ 调用 WebSocketNotificationService
  ↓
WebSocketManager 定向推送给订阅该设备的前端会话
  ↓
前端页面实时刷新
```

与此同时，还有一条“补偿链路”：

```text
定时任务 Scheduler
  ├─ 周期检查设备是否离线
  ├─ 周期采集系统指标
  ├─ 自动备份与清理过期数据
  └─ 为实时系统提供兜底和运维能力
```

这一章最重要的教学点是：

- MQTT 解决的是设备和后端之间的消息传递。
- WebSocket 解决的是后端和前端之间的实时推送。
- 定时任务解决的是巡检、补偿、清理和系统维护。

三者不是替代关系，而是协作关系。

## 2. MQTT通信设计

### 2.1 MQTT在本项目中的角色

在这个宿舍用电管理系统里，MQTT 主要用于设备侧通信，适合以下场景：

- 设备周期上报在线状态。
- 设备周期上报遥测数据。
- 设备返回命令执行应答。
- 设备上报告警或事件。

相比普通 HTTP，MQTT 更适合高频、轻量、双向感知较强的设备通信场景。

### 2.2 配置入口：MqttConfig

`MqttConfig` 负责读取配置并创建客户端，核心配置项包括：

- `mqtt.enabled`
- `mqtt.broker-url`
- `mqtt.client-id`
- `mqtt.username`
- `mqtt.password`
- `mqtt.topic-prefix`

它的职责可以概括为两件事：

1. 读取 Spring 配置文件中的 MQTT 参数。
2. 创建 `MqttClient` 并设置连接参数，如自动重连、心跳、超时等。

典型代码片段如下：

```java
public MqttClient createMqttClient() {
    if (!enabled) {
        return null;
    }

    MqttClient client = new MqttClient(brokerUrl, clientId);
    MqttConnectOptions options = new MqttConnectOptions();
    options.setAutomaticReconnect(true);
    options.setCleanSession(true);
    options.setConnectionTimeout(10);
    options.setKeepAliveInterval(60);
    client.connect(options);
    return client;
}
```

教学时可以借这段代码讲几个概念：

- **自动重连**：设备系统网络波动常见，不能靠人工恢复。
- **心跳保活**：用于检测连接是否仍然有效。
- **主题前缀**：有助于做多业务隔离和环境区分。

### 2.3 核心桥接类：MqttBridge

`MqttBridge` 是本项目设备通信的核心枢纽。它负责：

1. 启动时建立 MQTT 连接。
2. 订阅设备相关主题。
3. 收到消息后按类型分发处理。
4. 处理完成后通过 WebSocket 推送给前端。

它在初始化时会订阅两类主题模式：

```java
String[] singleLevelTopics = {
    prefix + "/+/status",
    prefix + "/+/telemetry",
    prefix + "/+/ack",
    prefix + "/+/event"
};

String[] twoLevelTopics = {
    prefix + "/+/+/status",
    prefix + "/+/+/telemetry",
    prefix + "/+/+/ack",
    prefix + "/+/+/event"
};
```

这说明当前项目支持两种设备主题层级：

- 单层设备标识，如 `dorm/device001/status`
- 双层设备标识，如 `dorm/buildingA/device001/status`

这是一个很适合教学的细节，因为它体现了后端已经考虑到设备 ID 不一定只有一个层级。

### 2.4 消息分发机制

MQTT 消息进入 `MqttBridge` 后，会先被解析主题，再进入统一分发：

```java
switch (msgType) {
    case "status":
        handleStatusMessage(deviceId, jsonNode);
        break;
    case "telemetry":
        handleTelemetryMessage(deviceId, jsonNode);
        break;
    case "ack":
        handleAckMessage(deviceId, jsonNode);
        break;
    case "event":
        handleEventMessage(deviceId, jsonNode);
        break;
}
```

这种写法非常适合教学，因为它让学生立刻明白：同一个传输通道可以承载多种业务消息，但落地处理逻辑必须分开。

### 2.5 状态消息处理

`handleStatusMessage()` 主要做了这些事：

1. 更新 `Device` 在线状态和最后活跃时间。
2. 更新 `StripStatus` 当前插排状态。
3. 写入 `DeviceStatusHistory` 历史记录。
4. 通过 `WebSocketNotificationService.broadcastDeviceStatus()` 推送给前端。

这段逻辑很适合拿来说明“设备消息不是只转发一下就结束”，而是会影响多个数据对象：

- 当前状态表
- 历史记录表
- 前端实时界面

### 2.6 遥测消息处理

`handleTelemetryMessage()` 主要完成：

1. 从 MQTT 载荷中提取功率、电压、电流。
2. 写入 `Telemetry` 表。
3. 调用 `AlertService` 检查是否越界。
4. 更新设备最后活跃时间。
5. 向前端广播遥测消息。

这正好对应设备后端的一个经典处理模板：

```text
接收消息 → 数据落库 → 规则检查 → 更新状态 → 实时通知
```

### 2.7 命令应答与事件消息

当前项目把命令执行结果和设备事件也纳入 MQTT：

- `ack` 消息进入 `handleAckMessage()`
  - 更新命令状态
  - 广播命令执行结果
- `event` 消息进入 `handleEventMessage()`
  - 广播设备告警或事件

这让设备控制流程形成闭环：

```text
后端下发命令
  ↓
设备执行
  ↓
设备通过 ack 返回结果
  ↓
后端更新命令状态并通知前端
```

### 2.8 命令下发

`MqttBridge` 也负责向设备发布命令：

```java
public boolean publishCommand(String deviceId, Object payload) {
    String topic = mqttConfig.buildTopic(deviceId, "cmd");
    String jsonPayload = objectMapper.writeValueAsString(payload);
    MqttMessage message = new MqttMessage(jsonPayload.getBytes());
    message.setQos(1);
    mqttClient.publish(topic, message);
    return true;
}
```

这里可以讲两个工程点：

- 发布前会先检查 MQTT 是否连接。
- 当前 QoS 设为 1，表示“至少一次”投递语义。

## 3. WebSocket实时推送设计

### 3.1 为什么还需要 WebSocket

很多学生第一次接触设备后端时会问：已经有 MQTT 了，为什么还要 WebSocket？

答案很简单，因为两个协议面向的对象不同：

- MQTT 面向设备侧接入。
- WebSocket 面向浏览器或前端管理端。

浏览器并不是当前项目里 MQTT 的直接消费端，因此后端需要把设备消息“转译”为前端可直接订阅的实时通道。

### 3.2 配置入口：WebSocketConfig

`WebSocketConfig` 使用 `@EnableWebSocket` 开启 WebSocket 支持，并注册 `/ws` 端点：

```java
@Configuration
@EnableWebSocket
public class WebSocketConfig implements WebSocketConfigurer {
    @Override
    public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
        registry.addHandler(new com.dormpower.websocket.WebSocketHandler(), "/ws")
                .setAllowedOrigins("*");
    }
}
```

教学时要指出两个事实：

1. 当前实现已经启用了 WebSocket。
2. 当前跨域是全开放的，代码里也写了后续需要收紧域名限制的提醒。

所以这里既能讲功能，也能顺带讲安全治理。

### 3.3 连接处理：WebSocketHandler

`WebSocketHandler` 负责处理前端连接生命周期和客户端发来的控制消息，包括：

- 建立连接时加入会话管理器。
- 关闭连接时清理订阅关系。
- 处理 `subscribe`、`unsubscribe`、`ping` 三种文本消息。

连接建立时会返回一条连接成功消息：

```java
message.put("type", "connection");
message.put("status", "connected");
message.put("sessionId", session.getId());
```

这很适合教学里引导学生理解：WebSocket 不只是“服务器推消息”，它同样可以有简单协议层。

### 3.4 订阅模型：WebSocketManager

`WebSocketManager` 是实时订阅的核心，它维护了三类数据结构：

1. 所有 WebSocket 会话集合。
2. 每个会话订阅了哪些设备。
3. 每个设备被哪些会话订阅。

这使得系统可以做“按设备定向推送”，而不是把所有数据都广播给所有客户端。

核心思路如下：

```text
session -> subscribed device ids
device id -> subscribed sessions
```

这是一种非常常见的实时系统设计模式，适合学生以后迁移到聊天、订单、告警中心等场景。

### 3.5 推送服务：WebSocketNotificationService

为了不让业务代码直接操作会话对象，项目又封装了一层 `WebSocketNotificationService`。它负责把业务消息组装成统一 JSON 结构，再交给 `WebSocketManager` 发送。

当前已经封装的消息类型包括：

- 设备状态更新
- 遥测更新
- 命令执行结果
- 异常告警
- 系统广播

例如，广播遥测数据时会组装：

```java
message.put("type", "TELEMETRY");
message.put("deviceId", deviceId);
message.put("payload", telemetry);
message.put("timestamp", System.currentTimeMillis());
```

这里值得教学强调的一点是：**实时推送也应有统一消息协议，而不是每个地方随手拼字符串。**

### 3.6 从 MQTT 到 WebSocket 的联动

当前项目里，MQTT 与 WebSocket 的关系非常清晰：

- `MqttBridge` 接收设备消息。
- `WebSocketNotificationService` 负责把处理后的数据推给前端。
- `WebSocketManager` 负责把消息送到正确的订阅者。

可以用一条典型链路帮助学生理解：

```text
设备上报 telemetry
  ↓
MqttBridge.handleTelemetryMessage()
  ↓
TelemetryRepository.save()
  ↓
AlertService.checkAndGenerateAlert()
  ↓
WebSocketNotificationService.broadcastTelemetry()
  ↓
WebSocketManager.sendToDeviceSubscribers(deviceId, message)
  ↓
前端收到实时更新
```

这条链路是第四章最关键的教学主线。

## 4. 定时任务与状态补偿

### 4.1 为什么实时系统还需要定时任务

很多初学者会误以为：既然设备能主动上报，那就不需要定时任务了。

这是不准确的。设备系统的现实情况是：

- 设备可能掉线。
- 消息可能丢失。
- 历史数据需要清理。
- 系统运维动作需要按周期执行。

所以定时任务不是实时通信的替代品，而是它的补偿和运维保障机制。

### 4.2 设备在线巡检：DeviceStatusMonitorScheduler

这个类每 30 秒检查一次设备是否离线：

```java
@Scheduled(fixedRate = 30000)
public void checkDeviceStatus() {
    List<Device> devices = deviceRepository.findAll();
    long now = System.currentTimeMillis() / 1000;
    boolean shouldBeOnline = (now - device.getLastSeenTs()) < OFFLINE_THRESHOLD_SECONDS;
}
```

它的主要作用是：

1. 根据 `lastSeenTs` 判断设备是否超时未上报。
2. 自动把设备状态改成离线或上线。
3. 写系统日志。
4. 发送离线告警通知。

这说明当前项目的“在线状态”并不完全依赖设备自己声明，还结合了服务端超时判断。

### 4.3 系统指标与运维任务：SystemMetricsScheduler

`SystemMetricsScheduler` 负责更偏运维侧的周期工作，包括：

- 每 5 分钟采集系统指标。
- 每天凌晨 2 点自动备份数据库。
- 每天凌晨 3 点清理过期日志、备份和监控指标。

这部分能让学生理解后端项目的另一面：一个成熟系统不仅要处理业务，还要持续维护自身运行状态。

### 4.4 当前实现中的一个重要现状

代码中已经存在多个 `@Scheduled` 标注的方法，例如：

- `DeviceStatusMonitorScheduler`
- `SystemMetricsScheduler`
- 以及其他服务里的定时逻辑

但是在当前已检索的代码范围内，没有找到显式的 `@EnableScheduling`。

这意味着教学表述应保持严谨：

- 可以确定：项目中已经设计并编写了定时任务类。
- 需要进一步核实：运行时是否在其他配置、依赖或外部启动方式中开启了调度支持。

课堂上可以把这点作为一次“代码审计训练”：**看到注解存在，不等于运行行为已完全打通。**

## 5. 当前实现特点与演进建议

### 5.1 当前实现的优点

基于现有代码，可以总结出几个很适合教学肯定的优点：

1. MQTT 与 WebSocket 分工清楚，没有混成一个通道。
2. `MqttBridge` 把状态、遥测、应答、事件做了明确分发。
3. WebSocket 使用“按设备订阅”而不是无差别全量广播。
4. 设备消息处理后会同时落库与推送，兼顾历史查询与实时展示。
5. 定时任务覆盖了离线巡检、监控采集、备份和清理。

### 5.2 当前实现的工程提醒

教学时也要诚实指出一些当前实现的工程现状：

1. `WebSocketConfig` 当前对跨域使用 `setAllowedOrigins("*")`，生产环境应收紧。
2. `WebSocketHandler` 是直接 `new` 出来的，而不是通过 Spring 注入 Bean，这种写法简单但扩展性一般。
3. `WebSocketManager` 使用手写单例管理连接，结构直观，但后续若要扩展认证、分布式推送、集群共享会话，会遇到上限。
4. `MqttBridge` 当前把解析、持久化、告警和推送都集中在一个桥接类里，教学上容易理解，但后续可以继续拆分职责。
5. 定时任务启用点当前未检索到，文档和课堂中都不应直接假定它已经完全生效。

### 5.3 推荐演进方向

如果后续继续迭代，这一部分可以作为高级教学讨论：

1. 用 Spring 管理的 WebSocket Handler Bean 代替直接 `new` 实例。
2. 将 MQTT 消息处理按领域拆成独立处理器，如状态处理器、遥测处理器、命令应答处理器。
3. 为 WebSocket 接入认证和更细粒度的订阅权限控制。
4. 如果要支持多实例部署，可考虑引入 Redis Pub/Sub 或消息总线做跨节点推送。
5. 明确补齐调度启用配置，并在启动流程中做可观测性检查。

## 6. 本章小结

### 关键知识点

1. MQTT 负责设备与后端之间的消息通信。
2. WebSocket 负责后端与前端之间的实时推送。
3. `MqttBridge` 是设备消息进入业务系统后的核心桥接点。
4. `WebSocketManager` 通过订阅关系实现按设备定向推送。
5. 定时任务负责离线巡检、指标采集、备份和清理，是实时系统的重要补充机制。

### 实践建议

1. 让学生手画一条 `telemetry` 消息从设备到前端的调用链。
2. 让学生比较 `broadcast()` 和 `sendToDeviceSubscribers()` 的区别，理解“全量广播”和“定向推送”的成本差异。
3. 让学生思考如果设备数量从几十台扩展到几万台，当前单机内存订阅管理会遇到什么问题。
4. 让学生检查项目中调度启用点，训练“代码存在不等于功能已生效”的工程判断能力。

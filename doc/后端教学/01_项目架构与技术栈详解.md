### 1.2 分层架构说明

本项目采用经典的**四层架构**，每层职责明确：

| 层级             | 职责                             | 关键注解          | 示例类                  |
| ---------------- | -------------------------------- | ----------------- | ----------------------- |
| **Controller层** | 接收HTTP请求，参数验证，返回响应 | `@RestController` | `AuthController`        |
| **Service层**    | 核心业务逻辑，事务管理           | `@Service`        | `AuthService`           |
| **Repository层** | 数据库操作，CRUD                 | `@Repository`     | `UserAccountRepository` |
| **Entity层**     | 数据模型，ORM映射                | `@Entity`         | `UserAccount`           |

### 1.3 请求处理流程

以用户登录为例，一个完整的请求会经历以下步骤：

```
客户端发送 POST /api/auth/login
    ↓
1. CORS跨域检查
    ↓
2. Spring Security过滤器链
    ├─ JWT过滤器检查令牌
    └─ 授权检查
    ↓
3. AOP切面拦截
    ├─ 限流检查 (2次/秒)
    └─ 记录请求日志
    ↓
4. Controller层处理
    ├─ @Valid参数验证
    └─ 调用Service层
    ↓
5. Service层业务逻辑
    ├─ 查询用户数据
    ├─ 验证密码
    └─ 生成JWT令牌
    ↓
6. Repository层数据访问
    └─ JPA查询数据库
    ↓
7. 返回响应
    └─ JSON格式的用户信息和令牌
```

### 2.5 为什么选择这些技术？

#### Spring Boot 3.2.3

```java
// Spring Boot的优势：自动配置
@SpringBootApplication  // 一个注解启动整个应用
public class DormPowerApplication {
    public static void main(String[] args) {
        SpringApplication.run(DormPowerApplication.class, args);
    }
}
```

- **自动配置**：根据依赖自动配置Bean
- **起步依赖**：一个依赖引入相关组件
- **内嵌服务器**：内置Tomcat，无需部署WAR包
- **生产就绪**：健康检查、指标监控

#### Java 17 LTS

```java
// Java 17 新特性示例
public record UserDTO(String username, String email, String role) {}

// 使用Record创建不可变数据对象
UserDTO user = new UserDTO("admin", "admin@dorm.local", "ADMIN");
System.out.println(user.username());  // 自动生成的getter方法
```

- **长期支持**：Oracle提供8年支持
- **性能提升**：G1垃圾收集器优化
- **新语法**：Records、Sealed Classes、Pattern Matching

#### PostgreSQL

- **开源免费**：无许可证费用
- **功能强大**：支持JSON、全文搜索
- **性能优秀**：复杂查询优化
- **扩展性强**：支持分区表、并行查询

#### 当前项目的缓存实现说明

- **声明式缓存**：通过 `@Cacheable`、`@CacheEvict` 配合 `ConcurrentMapCacheManager` 实现
- **轻量级手动缓存**：通过 `SimpleCacheService` 在低内存场景下做补充
- **教学要点**：不要把“常见方案”直接等同于“项目现状”，当前项目并未引入 Caffeine 或 Redis

## 4. 应用启动流程

### 4.1 启动类解析

```java
@SpringBootApplication
public class DormPowerApplication {

    public static void main(String[] args) {
        SpringApplication.run(DormPowerApplication.class, args);
    }
}
```

`@SpringBootApplication` 是一个组合注解，等价于：

```java
@SpringBootApplication =
    @SpringBootConfiguration +    // 配置类
    @EnableAutoConfiguration +    // 自动配置
    @ComponentScan                // 组件扫描
```

### 4.2 启动流程详解

```
1. 创建 SpringApplication 对象
   ├─ 推断应用类型（Servlet/Reactive）
   ├─ 加载初始化器 (ApplicationContextInitializer)
   └─ 加载监听器 (ApplicationListener)
    ↓
2. 执行 run() 方法
   ├─ 准备环境 (Environment)
   ├─ 打印Banner
   ├─ 创建ApplicationContext
   ├─ 准备上下文
   ├─ 刷新上下文（核心步骤）
   │   ├─ 扫描@Component注解的类
   │   ├─ 创建Bean定义
   │   ├─ 执行自动配置
   │   ├─ 依赖注入
   │   └─ 初始化Bean
   ├─ 发布事件
   └─ 返回ApplicationContext
    ↓
3. 应用启动完成
   └─ 监听HTTP请求
```

### 4.3 自动配置原理

Spring Boot通过`@EnableAutoConfiguration`实现自动配置：

```java
// 条件装配示例
@Configuration
@ConditionalOnClass(DataSource.class)  // 当类路径存在DataSource时生效
@ConditionalOnMissingBean(DataSource.class)  // 当容器中没有DataSource Bean时生效
public class DataSourceAutoConfiguration {

    @Bean
    @ConfigurationProperties(prefix = "spring.datasource")
    public DataSource dataSource() {
        return DataSourceBuilder.create().build();
    }
}
```

**常用条件注解**：

| 注解                           | 说明                       |
| ------------------------------ | -------------------------- |
| `@ConditionalOnClass`          | 类路径存在指定类时生效     |
| `@ConditionalOnMissingBean`    | 容器中不存在指定Bean时生效 |
| `@ConditionalOnProperty`       | 配置属性满足条件时生效     |
| `@ConditionalOnWebApplication` | Web应用时生效              |

### 5.2 环境配置文件

当前项目中能直接看到的环境文件包括：

- `application-dev.yml`：开发环境配置，适合联调与调试。
- `application-local.yml`：本地运行配置，通常用于个人机器开发。
- `application-prod.yml`：生产环境配置，用于正式部署。
- `application-low-memory.yml`：低内存场景配置，用于资源受限机器。

教学时建议强调两点：

1. **不要把环境差异写死在代码里**，而是通过不同 profile 或环境变量切换。
2. **敏感配置优先走环境变量**，当前项目通过 `spring.config.import` 支持从 `.env` 文件导入。

一个适合课堂讲解的观察点是：为什么 `server.port`、`MQTT_BROKER_URL`、`JWT_SECRET` 都采用 `${变量名:默认值}` 写法。它体现的是“同一份代码，多环境复用”的工程习惯。

### 5.3 配置注入方式

#### 方式1：`@Value` 注解

```java
@Service
public class AuthService {

    @Value("${admin.username:admin}")
    private String adminUsername;

    @Value("${security.jwt.expiration}")
    private long jwtExpiration;
}
```

项目中 `JwtUtil`、`MqttConfig`、`AuthService` 都大量使用了这种方式，优点是直接、上手快，缺点是配置项多时会变得分散。

#### 方式2：配置类集中封装

```java
@Configuration
public class MqttConfig {

    @Value("${mqtt.enabled:false}")
    private boolean enabled;

    @Value("${mqtt.broker-url}")
    private String brokerUrl;

    @Value("${mqtt.client-id}")
    private String clientId;
}
```

这里虽然没有使用 `@ConfigurationProperties`，但教学时可以顺势引导学生思考：当配置项继续变多时，是否应该收拢成更强类型的配置对象。

#### 方式3：环境变量与默认值

```java
@Value("${DATABASE_URL:jdbc:postgresql://localhost:5432/dormpower}")
private String databaseUrl;
```

工程上要让学生记住的不是语法本身，而是优先级思路：**环境变量 > 外部配置文件 > 项目默认值**。

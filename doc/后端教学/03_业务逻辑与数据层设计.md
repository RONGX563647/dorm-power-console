# 第三章：业务逻辑与数据层设计

> 本章进入后端项目最核心的日常开发区域：控制器怎么写、服务层做什么、仓储层如何查询、实体类如何建模，以及事务边界应该放在哪里。学完这一章，学生应能从“会看接口”进一步走到“会改业务”。

---

## 教学目标与安排

### 学习目标

1. 能说明 Controller、Service、Repository、Entity 四层在当前项目中的真实职责边界。
2. 能顺着真实代码读懂一次设备查询或遥测查询的调用链。
3. 能识别简单实体、关联实体和中间关联表在 JPA 中的表达方式。
4. 能理解方法名查询、分页查询、自定义 JPQL 查询的区别。
5. 能解释为什么有些业务要加 `@Transactional`，有些简单查询不一定显式加。

### 建议课堂安排

- **导入问题**：同样是“查数据”，为什么有的逻辑只写一层，有的逻辑要穿过四层？
- **讲解重点**：分层职责、实体建模、仓储查询、事务边界
- **代码走读顺序**：`DeviceController/DeviceService/DeviceRepository/Device` → `TelemetryController/TelemetryService/TelemetryRepository/Telemetry` → `RbacService/Role/Permission/UserRole`
- **实践任务**：让学生自己定位一个简单 CRUD 链路和一个复杂统计链路

## 1. 四层架构在项目中的落地

在这个项目里，四层架构不是抽象概念，而是非常明确的目录组织方式：

```text
HTTP请求
  ↓
Controller 接收参数、调用服务
  ↓
Service 承担业务逻辑与规则判断
  ↓
Repository 与数据库交互
  ↓
Entity 映射数据库表结构
```

最适合教学入门的链路是设备查询：

```text
GET /api/devices
  ↓
DeviceController.getDevices()
  ↓
DeviceService.getDevices()
  ↓
DeviceRepository.findAll()
  ↓
Device 实体映射数据库记录
```

更适合讲“业务层价值”的链路是遥测统计：

```text
GET /api/telemetry?device=device_001&range=24h
  ↓
TelemetryController.getTelemetry()
  ↓
TelemetryService.getTelemetry()
  ├─ 校验 range
  ├─ 计算时间窗口
  ├─ 查询仓储
  ├─ 补点/降采样/聚合
  └─ 组装返回结构
  ↓
TelemetryRepository.findByDeviceIdAndTsBetweenOrderByTsAsc(...)
```

教学时要让学生看到一个关键事实：**分层不是为了“看起来规范”，而是为了让每一层只处理自己最擅长的问题。**

## 2. Controller层设计

### 2.1 Controller层职责

Controller 层在当前项目里主要负责四件事：

1. 暴露 HTTP 接口。
2. 接收并校验请求参数。
3. 调用 Service 层。
4. 返回统一的 HTTP 响应。

它**不应该**承担复杂业务计算，也不应该直接拼接数据库查询逻辑。

### 2.2 典型示例：TelemetryController

```java
@RestController
@RequestMapping("/api")
public class TelemetryController {

    @Autowired
    private TelemetryService telemetryService;

    @GetMapping("/telemetry")
    public ResponseEntity<?> getTelemetry(
            @RequestParam String device,
            @RequestParam String range) {

        if (!range.matches("^(60s|24h|7d|30d)$")) {
            throw new BusinessException("range is invalid, must be one of: 60s, 24h, 7d, 30d");
        }

        List<Map<String, Object>> telemetryData = telemetryService.getTelemetry(device, range);
        return ResponseEntity.ok(telemetryData);
    }
}
```

这段代码很适合课堂上强调三点：

- **接口入口清晰**：学生可以很快知道 URL、参数和返回值在哪里定义。
- **参数校验前置**：不合法请求先挡在 Controller 层。
- **业务下沉**：真正的数据处理交给 `TelemetryService`。

### 2.3 Controller层常见写法总结

当前项目中常见的 Controller 特征包括：

- `@RestController` + `@RequestMapping` 组合定义 REST 接口。
- `@GetMapping`、`@PostMapping`、`@PutMapping`、`@DeleteMapping` 区分 HTTP 动词。
- `@RequestParam`、`@PathVariable`、`@RequestBody` 处理不同参数来源。
- OpenAPI 注解用于描述接口文档。
- 安全、限流、审计等横切逻辑通常通过 AOP 或 Security 统一处理，而不是散落在每个 Controller 方法里。

### 2.4 教学提示

学生最容易犯的错误是把 Controller 写成“大杂烩”：

- 在 Controller 中直接查多个 Repository。
- 在 Controller 中写复杂循环、统计和组装逻辑。
- 在 Controller 中处理事务。

课堂上要反复强调：**Controller 是接口层，不是业务中枢。**

## 3. Service层设计

### 3.1 Service层职责

Service 层是业务逻辑的核心。它在本项目里主要承担：

1. 业务规则判断。
2. 多个仓储或多个实体之间的协调。
3. 数据聚合、转换、裁剪与补全。
4. 抛出业务异常。
5. 在需要时声明事务边界。

### 3.2 典型示例一：DeviceService

```java
@Service
public class DeviceService {

    @Autowired
    private DeviceRepository deviceRepository;

    @Cacheable(value = "devices", unless = "#result == null || #result.isEmpty()")
    public List<Map<String, Object>> getDevices() {
        List<Device> devices = deviceRepository.findAll();
        List<Map<String, Object>> result = new ArrayList<>();

        for (Device device : devices) {
            Map<String, Object> deviceMap = new HashMap<>();
            deviceMap.put("id", device.getId());
            deviceMap.put("name", device.getName());
            deviceMap.put("room", device.getRoom());
            deviceMap.put("online", device.isOnline());
            deviceMap.put("lastSeen", device.getLastSeenTs());
            result.add(deviceMap);
        }
        return result;
    }
}
```

这里很适合讲“为什么 Service 不只是仓储转发器”：

- `Repository.findAll()` 拿到的是实体对象。
- 真正返回给前端的却是裁剪后的结构。
- 缓存也放在 Service 层，而不是 Controller 层。

也就是说，Service 层在这里完成了**查询结果重组**和**性能优化接入**。

### 3.3 典型示例二：TelemetryService

```java
@Service
public class TelemetryService {

    @Autowired
    private TelemetryRepository telemetryRepository;

    @Cacheable(value = "telemetry", key = "#deviceId + '_' + #range")
    public List<Map<String, Object>> getTelemetry(String deviceId, String range) {
        RangeConfig config = RANGE_CONFIG.get(range);
        if (config == null) {
            throw new BusinessException("Invalid range: " + range);
        }

        long nowTs = System.currentTimeMillis() / 1000;
        long startTs = nowTs - (long) (config.points - 1) * config.step;

        List<Telemetry> rows = telemetryRepository.findByDeviceIdAndTsBetweenOrderByTsAsc(deviceId, startTs, nowTs);

        if (rows.isEmpty()) {
            return Collections.emptyList();
        }

        return processNon60sRange(rows, config.points);
    }
}
```

这类 Service 更能体现“业务层不是 CRUD 包装层”，因为它做了：

- 范围配置选择。
- 时间窗口计算。
- 仓储调用。
- 数据点补齐或压缩。
- 统计口径统一。

教学时可以明确告诉学生：**凡是涉及“规则”“口径”“算法”“数据整形”的地方，都优先考虑放在 Service 层。**

### 3.4 Service层设计要点

当前项目里的 Service 大致可以分成三类：

| 类型         | 示例                            | 典型职责                               |
| ------------ | ------------------------------- | -------------------------------------- |
| 简单业务服务 | `DeviceService`                 | CRUD + 缓存 + 基础异常处理             |
| 聚合计算服务 | `TelemetryService`              | 时间范围换算、统计、数据重采样         |
| 复杂领域服务 | `RbacService`、`BillingService` | 多表协调、权限分配、费用计算、事务处理 |

## 4. Repository层设计

### 4.1 Repository层职责

Repository 层的目标很明确：**与数据库交互**。

它主要负责：

1. 根据实体类型执行持久化操作。
2. 通过方法名派生查询。
3. 在必要时使用 `@Query` 编写自定义 JPQL。
4. 支持分页、排序和组合条件查询。

### 4.2 典型示例：DeviceRepository

```java
@Repository
public interface DeviceRepository extends JpaRepository<Device, String> {

    List<Device> findByRoom(String room);

}
```

这是最基础的 Spring Data JPA 仓储接口写法。教学时要让学生理解：

- `JpaRepository<Device, String>` 说明实体是 `Device`，主键类型是 `String`。
- `findByRoom` 不是手写实现，而是 Spring Data JPA 根据方法名自动生成查询。
- 因为继承了 `JpaRepository`，所以 `findAll()`、`findById()`、`save()`、`deleteById()` 都已经可用。

### 4.3 分页与时间范围查询：TelemetryRepository

```java
@Repository
public interface TelemetryRepository extends JpaRepository<Telemetry, Long> {

    List<Telemetry> findByDeviceIdAndTsBetweenOrderByTsAsc(String deviceId, long startTs, long endTs);

    Page<Telemetry> findByDeviceIdOrderByTsDesc(String deviceId, Pageable pageable);

    Page<Telemetry> findByDeviceIdAndTsBetweenOrderByTsAsc(String deviceId, long startTs, long endTs, Pageable pageable);
}
```

这组方法非常适合用来教学“同一实体，不同查询粒度”：

- **列表查询**：适合服务层自行聚合。
- **分页查询**：适合后台管理页面或大数据集浏览。
- **时间范围查询**：适合遥测和统计类业务。

### 4.4 自定义 JPQL：RoleRepository

```java
@Repository
public interface RoleRepository extends JpaRepository<Role, String> {

    Optional<Role> findByCode(String code);

    @Query("SELECT r FROM Role r JOIN r.permissions p WHERE p.code = :permissionCode")
    List<Role> findByPermissionCode(@Param("permissionCode") String permissionCode);
}
```

这里可以带学生理解 JPQL 的两个重点：

- 它查的是**实体关系**，不是直接写表名和字段名。
- 当方法名查询已经不够表达复杂关联时，再进入 `@Query`。

### 4.5 关联表仓储：UserRoleRepository

```java
@Repository
public interface UserRoleRepository extends JpaRepository<UserRole, UserRoleId> {

    List<UserRole> findByUsername(String username);

    Optional<UserRole> findByUsernameAndRoleId(String username, String roleId);

    void deleteByRoleId(String roleId);

    @Query("SELECT ur.roleId FROM UserRole ur WHERE ur.username = :username")
    List<String> findRoleIdsByUsername(@Param("username") String username);
}
```

这部分很适合讲“组合键 + 中间表”的概念，因为学生在普通 CRUD 项目里往往很少接触到它。

### 4.6 Repository层教学结论

让学生记住一个判断规则：

- **方法名能表达清楚**，优先用方法名查询。
- **需要跨实体关系或更复杂筛选**，再用 `@Query`。
- **分页场景**，优先返回 `Page<T>`。
- **不要把业务规则写进 Repository**。

## 5. Entity层与数据建模

### 5.1 简单实体：Device

```java
@Entity
@Table(name = "devices")
public class Device {

    @Id
    private String id;

    @NotNull
    private String name;

    @NotNull
    private String room;

    @NotNull
    private boolean online;

    @NotNull
    private long lastSeenTs;

    @NotNull
    private long createdAt;
}
```

这是最适合入门教学的实体：

- `@Entity` 表示它会被 JPA 托管。
- `@Table(name = "devices")` 明确数据库表名。
- `@Id` 指定主键字段。
- `@NotNull` 同时体现了数据约束意识。

这类实体通常对应一张单表，字段也比较平直，适合学生先建立“类映射表”的基本认知。

### 5.2 带索引的实体：UserAccount

```java
@Entity
@Table(name = "user_account", indexes = {
        @Index(name = "idx_email", columnList = "email", unique = true)
})
public class UserAccount {

    @Id
    private String username;

    @NotNull
    private String email;

    @NotNull
    private String passwordHash;

    @NotNull
    private String role;
}
```

这里适合引导学生理解两个建模意识：

1. **账号类实体要考虑唯一性和索引**。
2. **安全敏感字段不存明文密码，而存哈希值**。

### 5.3 多对多关系：Role 与 Permission

```java
@Entity
public class Role {

    @ManyToMany(fetch = FetchType.LAZY)
    @JoinTable(
        name = "role_permissions",
        joinColumns = @JoinColumn(name = "role_id"),
        inverseJoinColumns = @JoinColumn(name = "permission_id")
    )
    private Set<Permission> permissions;
}
```

```java
@Entity
public class Permission {

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "resource_id")
    private Resource resource;
}
```

这一部分最值得讲的是：JPA 不只是在映射单表，它还在描述**表与表之间的关系**。

当前项目里的 RBAC 建模就能很好展示三种关系：

- `Role` 和 `Permission`：多对多。
- `Permission` 和 `Resource`：多对一。
- `UserRole`：显式中间关联实体。

### 5.4 组合键关联实体：UserRole

```java
@Entity
@Table(name = "user_roles")
@IdClass(UserRoleId.class)
public class UserRole {

    @Id
    private String username;

    @Id
    private String roleId;

    @NotNull
    private long assignedAt;
}
```

这里很适合告诉学生：不是所有关联表都要“糊弄成一对多”。有些时候，中间表本身就有业务意义，比如：

- 谁被分配了哪个角色。
- 角色是什么时候分配的。
- 分配人是谁。

一旦中间表自身有业务字段，就很适合建成独立实体。

### 5.5 Entity层设计提醒

课堂上应避免学生把实体类写成“万能类”。当前项目提供了一个较好的反例参考：

- 实体类主要负责字段与关系映射。
- 大部分业务规则不塞进实体。
- 复杂计算放到 Service 层。

这能让数据模型保持稳定，也让业务逻辑更易维护。

## 6. 事务管理与边界控制

### 6.1 为什么需要事务

事务最核心的作用是保证一组操作要么一起成功，要么一起失败。

在当前项目里，简单查询服务并不一定显式标注事务，但复杂写操作已经在多个 Service 中使用了 `@Transactional`。这很适合拿来教学“事务不是到处乱加，而是用于关键一致性场景”。

### 6.2 真实示例一：RbacService

```java
@Transactional
public void deleteRole(String roleId) {
    Role role = roleRepository.findById(roleId)
            .orElseThrow(() -> new RuntimeException("Role not found: " + roleId));

    if (role.isSystem()) {
        throw new RuntimeException("Cannot delete system role: " + role.getCode());
    }

    userRoleRepository.deleteByRoleId(roleId);
    roleRepository.delete(role);
}
```

这段代码就是典型的事务场景：

- 先删用户角色关联。
- 再删角色本身。

如果中间任何一步失败，数据都不应该处于“删了一半”的状态，所以这里需要事务。

### 6.3 真实示例二：BillingService

```java
@Transactional
public RechargeRecord recharge(String roomId, double amount, String paymentMethod, String operator) {
    DormRoom room = dormRoomRepository.findById(roomId)
            .orElseThrow(() -> new RuntimeException("Room not found"));

    RoomBalance balance = roomBalanceRepository.findByRoomId(roomId)
            .orElseGet(() -> createRoomBalance(roomId));

    double balanceBefore = balance.getBalance();
    double balanceAfter = balanceBefore + amount;

    balance.setBalance(balanceAfter);
    roomBalanceRepository.save(balance);

    RechargeRecord record = new RechargeRecord();
    record.setRoomId(roomId);
    record.setAmount(amount);

    return rechargeRecordRepository.save(record);
}
```

这里同样不是单表操作，而是：

- 更新余额。
- 写入充值记录。

这是非常标准的“账务类事务”教学案例。

### 6.4 当前项目的事务现状

从代码现状看，可以归纳出一个很适合教学的观察：

- `DeviceService`、`TelemetryService` 这类简单查询或单仓储操作，没有普遍显式加 `@Transactional`。
- `RbacService`、`BillingService`、`StudentService` 等涉及多表一致性的服务，使用了 `@Transactional`。

这说明当前项目虽然不是“事务治理特别严格”的风格，但基本遵循了一个正确方向：**跨多个持久化动作的业务优先声明事务。**

### 6.5 事务教学结论

让学生先掌握下面这条判断原则就够用了：

1. **只读查询**：不一定要手动声明事务。
2. **单表简单写入**：可以暂不显式声明，但要知道背后仍有持久化上下文。
3. **多表写入、删除、余额或权限类操作**：优先加 `@Transactional`。

## 7. 本章小结

### 关键知识点

1. **Controller层**：负责接请求、校验参数、调用服务，不应承载复杂业务逻辑。
2. **Service层**：负责业务规则、数据聚合、异常抛出和事务边界控制。
3. **Repository层**：负责持久化访问，优先使用方法名查询，必要时使用 `@Query`。
4. **Entity层**：负责表结构与关系映射，既包括简单实体，也包括多对多和组合键关联实体。
5. **事务管理**：在 RBAC、计费、充值等多表一致性场景中尤为关键。

### 实践建议

1. 让学生自己画出 `DeviceController → DeviceService → DeviceRepository → Device` 的调用链。
2. 用 `TelemetryService` 讲“业务层为什么不能直接省掉”。
3. 用 `Role`、`Permission`、`UserRole` 三个实体讲清 JPA 关系映射。
4. 用 `RbacService.deleteRole()` 或 `BillingService.recharge()` 讲清事务边界应该放在哪一层。

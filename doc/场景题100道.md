### 题1：WebSocketManager使用双重检查锁实现单例模式

**背景描述**：在DormPower项目中，WebSocketManager使用双重检查锁（DCL）实现单例模式，用于管理所有WebSocket连接和订阅关系。

**问题描述**：分析WebSocketManager使用双重检查锁实现单例模式的线程安全性和潜在问题。

**难度等级**：★★★★

**考察知识点**：
- DCL单例模式实现
- volatile关键字语义
- 指令重排序问题
- 线程安全的单例创建

**参考答案要点**：
1. **问题分析**：
   - 双重检查锁（DCL）是一种常见的单例模式实现方式，通过两次检查来减少同步开销
   - 主要问题在于指令重排序可能导致对象在未完全初始化时被其他线程访问

2. **解决方案**：
   - 使用volatile关键字修饰单例实例，防止指令重排序
   - 确保对象完全初始化后再被其他线程访问

3. **代码示例**：
   ```java
   public class WebSocketManager {
       // 使用volatile关键字防止指令重排序
       private static volatile WebSocketManager instance;
       
       private WebSocketManager() {
           // 初始化操作
       }
       
       public static WebSocketManager getInstance() {
           if (instance == null) {
               synchronized (WebSocketManager.class) {
                   if (instance == null) {
                       instance = new WebSocketManager();
                   }
               }
           }
           return instance;
       }
   }
   ```

4. **最佳实践建议**：
   - 优先使用静态内部类单例模式，更简洁且线程安全
   - 如需使用DCL，必须使用volatile关键字
   - 考虑使用枚举单例，天然线程安全且防止反射和序列化攻击

**延伸思考**：
- 除了DCL，还有哪些单例模式实现方式？各自的优缺点是什么？
- 在Spring Boot环境中，如何通过容器管理单例？

### 题3：线程池大小计算逻辑

**背景描述**：MqttSimulatorService使用固定线程池Executors.newFixedThreadPool来模拟设备消息，需要确定合适的线程池大小。

**问题描述**：分析线程池大小的计算逻辑，如何根据任务类型（CPU密集型vs IO密集型）和系统资源来确定线程数。

**难度等级**：★★★

**考察知识点**：
- 线程池参数配置
- CPU密集型vs IO密集型任务特性
- 系统资源限制分析
- 线程池调优策略

**参考答案要点**：
1. **问题分析**：
   - CPU密集型任务：线程数应接近CPU核心数
   - IO密集型任务：线程数可以大于CPU核心数，通常为核心数的2-4倍
   - 系统内存限制：每个线程占用一定内存，需要考虑内存容量

2. **解决方案**：
   - CPU密集型：线程数 = CPU核心数 ± 1
   - IO密集型：线程数 = CPU核心数 × 2
   - 混合型任务：根据IO等待时间调整

3. **代码示例**：
   ```java
   // 获取CPU核心数
   int coreCount = Runtime.getRuntime().availableProcessors();
   
   // IO密集型任务线程池
   ExecutorService executorService = Executors.newFixedThreadPool(coreCount * 2);
   
   // 或者使用ThreadPoolExecutor自定义
   ThreadPoolExecutor executor = new ThreadPoolExecutor(
       coreCount,          // 核心线程数
       coreCount * 2,      // 最大线程数
       60L,                // 空闲线程存活时间
       TimeUnit.SECONDS,   // 时间单位
       new LinkedBlockingQueue<>(1000),  // 工作队列
       new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
   );
   ```

4. **最佳实践建议**：
   - 根据任务特性选择合适的线程池大小
   - 监控线程池的实际使用情况，动态调整
   - 合理设置队列大小，避免内存溢出
   - 选择合适的拒绝策略，确保系统稳定性

**延伸思考**：
- 如何监控线程池的运行状态？
- 线程池饱和时的处理策略有哪些？

   ```java
   public synchronized void incrementCount() {
       count++;
   }
   ```

4. **最佳实践建议**：
   - 优先使用原子类进行简单的计数和状态更新
   - 对于复杂的复合操作，考虑使用synchronized或ReentrantLock
   - 注意AtomicInteger可能存在的ABA问题
   - 对于高并发场景，考虑使用LongAdder以获得更好的性能

### 题5：多线程环境下的双向映射一致性

**背景描述**：WebSocketManager中使用sessionDeviceSubscriptions和deviceSubscribers两个Map来维护会话和设备的双向映射关系。

**问题描述**：分析多线程环境下如何保证这两个Map的一致性，以及如何处理复合操作的原子性。

**难度等级**：★★★★

**考察知识点**：
- 并发数据结构使用
- 双向映射一致性保证
- 复合操作原子性
- 线程安全的集合操作

**参考答案要点**：
1. **问题分析**：
   - 双向映射需要保证两个Map的操作原子性
   - 多线程环境下可能出现部分更新的情况
   - 需要确保添加和删除操作的一致性

2. **解决方案**：
   - 使用同步锁保证复合操作的原子性
   - 或者使用ConcurrentHashMap并确保操作的顺序性

3. **代码示例**：
   ```java
   // 使用同步锁保证一致性
   private final Map<WebSocketSession, Set<String>> sessionDeviceSubscriptions = new HashMap<>();
   private final Map<String, Set<WebSocketSession>> deviceSubscribers = new HashMap<>();
   
   public synchronized void subscribe(WebSocketSession session, String deviceId) {
       // 更新会话到设备的映射
       sessionDeviceSubscriptions.computeIfAbsent(session, k -> new HashSet<>()).add(deviceId);
       // 更新设备到会话的映射
       deviceSubscribers.computeIfAbsent(deviceId, k -> new HashSet<>()).add(session);
   }
   
   public synchronized void unsubscribe(WebSocketSession session, String deviceId) {
       // 从会话到设备的映射中移除
       Set<String> devices = sessionDeviceSubscriptions.get(session);
       if (devices != null) {
           devices.remove(deviceId);
           if (devices.isEmpty()) {
               sessionDeviceSubscriptions.remove(session);
           }
       }
       // 从设备到会话的映射中移除
       Set<WebSocketSession> sessions = deviceSubscribers.get(deviceId);
       if (sessions != null) {
           sessions.remove(session);
           if (sessions.isEmpty()) {
               deviceSubscribers.remove(deviceId);
           }
       }
   }
   ```

4. **最佳实践建议**：
   - 使用同步锁或ConcurrentHashMap保证操作原子性
   - 确保双向映射的添加和删除操作成对执行
   - 考虑使用不可变对象减少并发冲突
   - 定期清理无效的映射关系

**延伸思考**：
- 如何处理WebSocket会话关闭时的清理操作？
- 如何优化双向映射的内存使用？

### 题6：volatile boolean vs AtomicBoolean

**背景描述**：在模拟器任务中，使用volatile boolean running标志位来控制任务的运行状态。

- 原子性vs可见性

**参考答案要点**：
1. **问题分析**：

   - AtomicBoolean保证原子性，但在简单场景下可能过度设计
   - 对于简单的标志位控制，volatile足够

2. **解决方案**：
   - 对于简单的状态标志，使用volatile boolean
   - 对于需要原子操作的场景，使用AtomicBoolean

3. **代码示例**：
   ```java
   // 使用volatile boolean

       running = false;
   }
   
   public void run() {

           // 执行任务
       }
   }
   
   // 使用AtomicBoolean
   private final AtomicBoolean running = new AtomicBoolean(true);

       running.set(false);

           // 执行任务
       }
   }
   ```

4. **最佳实践建议**：
   - 对于简单的状态标志，优先使用volatile
   - 对于需要原子操作的场景，使用AtomicBoolean
   - 注意volatile不保证复合操作的原子性

   ```java
           String token = getTokenFromQuery(uri);

               token = request.getHeaders().getFirst("Authorization");

           if (query != null) {
   ```

   ```java
   public class MqttReconnectHandler implements MqttCallback {
       private MqttClient client;
       private int reconnectAttempts = 0;
       private final int MAX_RECONNECT_ATTEMPTS = 10;
       private final long BASE_RECONNECT_DELAY = 1000; // 1秒
       
       @Override
       public void connectionLost(Throwable cause) {
           logger.warn("MQTT connection lost: {}", cause.getMessage());
           
           if (reconnectAttempts < MAX_RECONNECT_ATTEMPTS) {
               // 计算指数退避延迟
               long delay = BASE_RECONNECT_DELAY * (1 << reconnectAttempts);
               // 最大延迟限制为30秒
               delay = Math.min(delay, 30000);
               
               logger.info("Attempting to reconnect in {}ms (attempt {}/{})", 
                   delay, reconnectAttempts + 1, MAX_RECONNECT_ATTEMPTS);
               
               // 延迟重连
               new Thread(() -> {
                   try {
                       Thread.sleep(delay);
                       reconnect();
                   } catch (InterruptedException e) {
                       Thread.currentThread().interrupt();
                   }
               }).start();
               
               reconnectAttempts++;
           } else {
               logger.error("Max reconnect attempts reached, giving up");
           }
       }
       
       private void reconnect() {
           try {
               if (!client.isConnected()) {
                   client.connect();
                   logger.info("MQTT reconnected successfully");
                   reconnectAttempts = 0; // 重置重连计数器
                   // 重新订阅主题
                   subscribeToTopics();
               }
           } catch (MqttException e) {
               logger.error("Failed to reconnect: {}", e.getMessage());
           }
       }
       
       private void subscribeToTopics() throws MqttException {
           // 重新订阅所有主题
           client.subscribe(new String[]{"dorm/+/status", "dorm/+/telemetry"}, 
               new int[]{1, 1});
       }
   }
   ```

   ```java
   public class TelemetryAggregator {

       private final TelemetryRepository telemetryRepository;

       @PostConstruct

               this::aggregateData, 1, 1, TimeUnit.MINUTES

           buffer.addData(data);

               String deviceId = entry.getKey();

           );
       }
   ```

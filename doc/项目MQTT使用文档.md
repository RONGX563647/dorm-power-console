## 1. 基础概念与原理

### 1.1 什么是MQTT？

MQTT（Message Queuing Telemetry Transport，消息队列遥测传输协议）是一种基于**发布/订阅模式**的轻量级通信协议，专为**低带宽、不稳定网络环境**下的物联网设备设计。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          MQTT协议特点                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  轻量级                                                           │   │
│  │  - 协议头部最小仅2字节                                            │   │
│  │  - 适合资源受限设备（传感器、嵌入式）                              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  发布/订阅模式                                                    │   │
│  │  - 解耦发布者和订阅者                                             │   │
│  │  - 一对多通信                                                     │   │
│  │  - 支持主题过滤                                                   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  可靠性保证                                                       │   │
│  │  - 三种QoS级别                                                    │   │
│  │  - 持久会话                                                       │   │
│  │  - 遗嘱消息                                                       │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  双向通信                                                         │   │
│  │  - 设备可发布消息                                                 │   │
│  │  - 设备可接收命令                                                 │   │
│  │  - 实时性强                                                       │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.3 核心组件

| 组件 | 说明 | 类比 |
|------|------|------|
| **Broker** | 消息代理服务器，负责转发消息 | 快递中转站 |
| **Client** | 客户端，可以是发布者或订阅者 | 发件人/收件人 |
| **Topic** | 主题，消息路由的地址 | 快递地址 |
| **Message** | 消息，传输的数据内容 | 快递包裹 |
| **QoS** | 服务质量等级，消息传递保证 | 快递服务等级 |

## 2. 核心特性与机制

### 2.1 QoS服务质量等级

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          QoS等级对比                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  QoS 0 - 最多一次（At Most Once）                                        │
│  ┌──────────┐     ┌──────────┐                                         │
│  │ Publisher │────>│  Broker  │  不确认，不重试                         │
│  │          │     │          │  适合：高频遥测数据                       │
│  └──────────┘     └──────────┘  性能最高，可靠性最低                     │
│                                                                         │
│  QoS 1 - 至少一次（At Least Once）                                       │
│  ┌──────────┐     ┌──────────┐                                         │
│  │ Publisher │────>│  Broker  │  确认机制，可能重复                      │
│  │          │<────│  PUBACK  │  适合：命令、状态数据                     │
│  └──────────┘     └──────────┘  性能中等，可靠性中等                     │
│                                                                         │
│  QoS 2 - 恰好一次（Exactly Once）                                        │
│  ┌──────────┐     ┌──────────┐                                         │
│  │ Publisher │────>│  Broker  │  四次握手，保证唯一                      │
│  │          │<────│  PUBREC  │  适合：关键业务数据                       │
│  │          │────>│  PUBREL  │  性能最低，可靠性最高                     │
│  │          │<────│  PUBCOMP │                                         │
│  └──────────┘     └──────────┘                                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 会话与持久化

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          会话机制                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Clean Session = true（清除会话）                                        │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  - 客户端断开后，Broker删除所有订阅信息                           │   │
│  │  - 重新连接时，需要重新订阅                                       │   │
│  │  - 不保留离线消息                                                 │   │
│  │  - 适合：临时客户端、不需要离线消息                                │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  Clean Session = false（持久会话）                                       │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  - 客户端断开后，Broker保留订阅信息                               │   │
│  │  - 重新连接时，自动恢复订阅                                       │   │
│  │  - 保留离线消息（QoS 1/2）                                        │   │
│  │  - 适合：需要接收离线消息的客户端                                  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.3 遗嘱消息（LWT）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          遗嘱消息机制                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  连接时设置遗嘱：                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │

│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  应用场景：                                                              │
│  ┌─────────────────────────────────────────────────────────────────┐   │

│  │  - 状态监控                                                       │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.4 保留消息（Retained Message）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          保留消息机制                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  特点：                                                                  │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  - Broker会保留该主题的最后一条保留消息                           │   │
│  │  - 新订阅者会立即收到保留消息                                     │   │
│  │  - 每个主题只能有一条保留消息                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  使用场景：                                                              │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  - 设备初始状态                                                   │   │
│  │  - 配置信息                                                       │   │
│  │  - 系统公告                                                       │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  代码示例：                                                              │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  MqttMessage message = new MqttMessage(payload);                 │   │
│  │  message.setRetained(true);  // 设置为保留消息                    │   │
│  │  client.publish("dorm/device_001/status", message);              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 4. Mosquitto实战

### 4.1 安装与配置

```bash
# Docker方式安装
docker run -d --name mosquitto \
  -p 1883:1883 \
  -p 9001:9001 \
  -v /path/to/config:/mosquitto/config \
  -v /path/to/data:/mosquitto/data \
  -v /path/to/log:/mosquitto/log \
  eclipse-mosquitto:2
```

### 4.2 配置文件详解

```
# /mosquitto/config/mosquitto.conf

# 监听端口
listener 1883

# 允许匿名连接（生产环境建议false）
allow_anonymous true

# 持久化配置
persistence true
persistence_location /mosquitto/data/

# 日志配置
log_dest file /mosquitto/log/mosquitto.log
log_type all

# 认证配置（可选）
# password_file /mosquitto/config/passwd
# acl_file /mosquitto/config/acl

# WebSocket支持
listener 9001
protocol websockets
```

### 4.3 命令行工具

```bash
# 订阅主题
mosquitto_sub -h localhost -p 1883 -t "dorm/#" -v

# 发布消息
mosquitto_pub -h localhost -p 1883 -t "dorm/device_001/status" \
  -m '{"online":true,"power_w":150.5}'

# 带用户名密码
mosquitto_sub -h localhost -p 1883 -u admin -P password -t "dorm/#"

# 保留消息
mosquitto_pub -h localhost -p 1883 -t "dorm/device_001/status" \
  -m '{"online":true}' -r
```

## 5. 最佳实践与优化

### 5.1 主题设计最佳实践

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          主题设计规范                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  推荐格式：                                                              │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  {prefix}/{category}/{resource}/{action}                        │   │
│  │                                                                  │   │
│  │  示例：                                                          │   │
│  │  dorm/device/device_001/status    设备状态                       │   │
│  │  dorm/device/device_001/cmd       设备命令                       │   │
│  │  dorm/device/device_001/telemetry 遥测数据                       │   │
│  │  dorm/device/device_001/ack       命令确认                       │   │
│  │  dorm/device/device_001/event     事件上报                       │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  注意事项：                                                              │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  1. 不要以 / 开头（会导致额外的空层级）                           │   │
│  │  2. 不要使用中文或特殊字符                                        │   │
│  │  3. 层级不宜过深（建议不超过5层）                                  │   │
│  │  4. 使用有意义的命名                                              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.2 QoS选择策略

| 数据类型 | 推荐QoS | 原因 |
|----------|---------|------|
| 高频遥测数据 | QoS 0 | 数据量大，丢失少量可接受 |
| 设备状态 | QoS 1 | 需要保证送达，允许少量重复 |
| 控制命令 | QoS 1 | 需要保证送达，业务层去重 |
| 关键告警 | QoS 1/2 | 必须送达 |
| 配置同步 | QoS 1 + Retain | 需要新设备立即获取 |

## 核心代码实现

### 1. MQTT配置类

```java
@Configuration
public class MqttConfig {

    @Value("${mqtt.enabled:false}")
    private boolean enabled;

    @Value("${mqtt.broker-url}")
    private String brokerUrl;

    @Value("${mqtt.client-id}")
    private String clientId;

    @Value("${mqtt.username}")
    private String username;

    @Value("${mqtt.password}")
    private String password;

    @Value("${mqtt.topic-prefix:dorm}")
    private String topicPrefix;

    /**
     * 创建MQTT客户端实例
     * 配置自动重连、心跳间隔等参数
     */
    public MqttClient createMqttClient() {
        if (!enabled) {
            return null;
        }

        try {
            MqttClient client = new MqttClient(brokerUrl, clientId);
            MqttConnectOptions options = new MqttConnectOptions();
            
            // 设置认证信息
            if (username != null && !username.isEmpty()) {
                options.setUserName(username);
                options.setPassword(password.toCharArray());
            }
            
            // 连接参数配置
            options.setAutomaticReconnect(true);  // 自动重连
            options.setCleanSession(true);        // 清除会话
            options.setConnectionTimeout(10);     // 连接超时10秒
            options.setKeepAliveInterval(60);     // 心跳间隔60秒
            
            client.connect(options);
            return client;
        } catch (MqttException e) {
            System.err.println("Failed to connect to MQTT broker: " + e.getMessage());
            return null;
        }
    }

    /**
     * 构建主题
     * 格式: {prefix}/{deviceId}/{suffix}
     */
    public String buildTopic(String deviceId, String suffix) {
        return topicPrefix + "/" + deviceId + "/" + suffix;
    }
}
```

### 2. MQTT桥接类（核心）

```java
@Component
public class MqttBridge {

    @Autowired
    private MqttConfig mqttConfig;

    @Autowired
    private WebSocketNotificationService webSocketNotificationService;

    @Autowired
    private TelemetryProducer telemetryProducer;  // Kafka生产者

    @Value("${kafka.enabled:true}")
    private boolean kafkaEnabled;

    private MqttClient mqttClient;
    private boolean connected = false;

    /**
     * 初始化：连接Broker并订阅主题
     */
    @PostConstruct
    public void init() {
        if (!mqttConfig.isEnabled()) {
            return;
        }

        try {
            mqttClient = mqttConfig.createMqttClient();
            if (mqttClient != null) {
                setupCallback();
                subscribeTopics();
                connected = true;
            }
        } catch (Exception e) {
            System.err.println("Failed to initialize MQTT bridge: " + e.getMessage());
        }
    }

    /**
     * 设置消息回调
     */
    private void setupCallback() {
        mqttClient.setCallback(new MqttCallback() {
            @Override
            public void connectionLost(Throwable cause) {
                connected = false;
                System.err.println("MQTT connection lost: " + cause.getMessage());
            }

            @Override
            public void messageArrived(String topic, MqttMessage message) {
                String payload = new String(message.getPayload());
                handleMessage(topic, payload);
            }

            @Override
            public void deliveryComplete(IMqttDeliveryToken token) {
                System.out.println("MQTT message delivery complete");
            }
        });
    }

    /**
     * 订阅主题
     */
    private void subscribeTopics() throws MqttException {
        String prefix = mqttConfig.getTopicPrefix();
        
        // 订阅单级和双级主题
        String[] topics = {
            prefix + "/+/status",
            prefix + "/+/telemetry",
            prefix + "/+/ack",
            prefix + "/+/event"
        };

        for (String topic : topics) {
            mqttClient.subscribe(topic, 1);
        }
    }

    /**
     * 处理接收到的消息
     */
    private void handleMessage(String topic, String payload) {
        try {
            TopicInfo topicInfo = parseTopic(topic);
            JsonNode jsonNode = objectMapper.readTree(payload);

            switch (topicInfo.msgType) {
                case "status":
                    handleStatusMessage(topicInfo.deviceId, jsonNode);
                    break;
                case "telemetry":
                    handleTelemetryMessage(topicInfo.deviceId, jsonNode);
                    break;
                case "ack":
                    handleAckMessage(topicInfo.deviceId, jsonNode);
                    break;
                case "event":
                    handleEventMessage(topicInfo.deviceId, jsonNode);
                    break;
            }
        } catch (Exception e) {
            System.err.println("Error handling MQTT message: " + e.getMessage());
        }
    }

    /**
     * 发布命令到设备
     */
    public boolean publishCommand(String deviceId, Object payload) {
        if (!connected || mqttClient == null || !mqttClient.isConnected()) {
            return false;
        }

        try {
            String topic = mqttConfig.buildTopic(deviceId, "cmd");
            String jsonPayload = objectMapper.writeValueAsString(payload);
            
            MqttMessage message = new MqttMessage(jsonPayload.getBytes());
            message.setQos(1);  // 命令使用QoS 1
            mqttClient.publish(topic, message);
            
            return true;
        } catch (Exception e) {
            return false;
        }
    }

    /**
     * 处理状态消息
     * 优先使用Kafka异步处理，提高吞吐量
     */
    private void handleStatusMessage(String deviceId, JsonNode payload) {
        if (kafkaEnabled && telemetryProducer != null) {
            // 发送到Kafka，异步处理
            telemetryProducer.sendDeviceStatus(deviceId, ...);
        } else {
            // 降级：直接处理
            processStatusDirectly(deviceId, payload);
        }
    }

    /**
     * 销毁：断开连接
     */
    @PreDestroy
    public void destroy() {
        if (mqttClient != null && mqttClient.isConnected()) {
            try {
                mqttClient.disconnect();
                mqttClient.close();
            } catch (MqttException e) {
                System.err.println("Error stopping MQTT bridge: " + e.getMessage());
            }
        }
    }
}
```

```yaml
      - mosquitto
```

## 面试常见问题与回答

### Q1: 为什么选择MQTT而不是HTTP？

**回答要点：**

1. **协议开销**：MQTT协议头最小仅2字节，HTTP请求头通常几百字节，对于资源受限的IoT设备更友好
2. **实时性**：MQTT是长连接，消息实时推送；HTTP需要轮询，延迟高且浪费资源
3. **双向通信**：MQTT一条连接支持双向通信；HTTP需要客户端轮询或WebSocket
4. **网络适应性**：MQTT专为不稳定网络设计，有QoS保证；HTTP在网络不稳定时容易失败

**项目中的体现：**
- 智能插座每秒上报遥测数据，使用MQTT QoS 0，开销极小
- 远程控制命令使用MQTT QoS 1，保证可靠性
- 设备异常断开时通过遗嘱消息自动通知后端

### Q2: MQTT的QoS级别如何选择？

**回答要点：**

| 场景 | QoS级别 | 原因 |
|------|---------|------|
| 高频遥测数据 | QoS 0 | 数据量大，丢失少量可接受，追求吞吐量 |
| 设备状态 | QoS 1 | 需要保证送达，允许业务层去重 |
| 控制命令 | QoS 1 | 必须送达，业务层通过cmdId去重 |
| 关键告警 | QoS 1/2 | 必须送达 |

**项目中的体现：**
```java
// 遥测数据：QoS 0
mqttClient.subscribe(prefix + "/+/telemetry", 0);

// 命令下发：QoS 1
message.setQos(1);
mqttClient.publish(topic, message);
```

### Q3: 如何保证消息不丢失？

**回答要点：**

1. **QoS 1保证**：至少送达一次，Broker会确认消息
2. **持久会话**：Clean Session = false，离线消息会被保留
3. **业务层确认**：命令下发后等待设备ACK，超时标记失败
4. **数据持久化**：消息处理后写入数据库

**项目中的体现：**
```java
// 命令下发后等待ACK
public boolean publishCommand(String deviceId, Object payload) {
    // 发布命令
    mqttClient.publish(topic, message);
    // 设备返回ACK后更新状态
}

// 超时处理
@Scheduled(fixedRate = 10000)
public void markTimeouts() {
    // 标记超时的命令为failed
}
```

### Q4: 如何处理MQTT消息重复？

**回答要点：**

1. **业务层去重**：每条消息携带唯一ID，消费端缓存已处理的ID
2. **幂等设计**：操作设计为幂等，重复执行结果相同
3. **数据库约束**：利用唯一索引防止重复写入

**项目中的体现：**
```java
// 命令去重：通过cmdId唯一标识
public void handleAckMessage(String deviceId, JsonNode payload) {
    String cmdId = payload.get("cmdId").asText();
    // 更新命令状态（幂等操作）
    commandService.updateCommandState(cmdId, status, message);
}
```

### Q5: MQTT与Kafka如何配合使用？

**回答要点：**

MQTT和Kafka定位不同：
- **MQTT**：设备接入层，处理设备连接、实时通信
- **Kafka**：数据管道层，处理数据持久化、异步处理

**配合方式：**
```
设备 → MQTT Broker → 后端服务 → Kafka → 消费者 → 数据库
```

**项目中的体现：**
```java
private void handleTelemetryMessage(String deviceId, JsonNode payload) {
    if (kafkaEnabled && telemetryProducer != null) {
        // 发送到Kafka，异步处理
        telemetryProducer.sendTelemetry(deviceId, ts, powerW, voltageV, currentA);
    } else {
        // 降级：直接写入数据库
        telemetryRepository.save(telemetry);
    }
}
```

**优势：**
1. MQTT专注设备通信，Kafka专注数据处理
2. Kafka消费者可以批量处理，提高吞吐量
3. Kafka支持多实例部署，实现水平扩展

### Q6: 如何实现设备离线检测？

**回答要点：**

1. **遗嘱消息（LWT）**：设备连接时设置遗嘱，异常断开时自动发送
2. **心跳检测**：通过Keep Alive机制检测连接状态
3. **超时判断**：超过一定时间未收到消息，标记离线

**项目中的体现：**
```java
// 连接时设置遗嘱
MqttConnectOptions options = new MqttConnectOptions();
options.setWill("dorm/device_001/lwt", 
                "{\"online\":false}".getBytes(),
                1, false);

// 后端订阅遗嘱主题
mqttClient.subscribe(prefix + "/+/lwt", 1);
```

### Q7: 项目中MQTT的性能优化有哪些？

**回答要点：**

1. **连接复用**：使用单一MQTT连接，避免频繁创建销毁
2. **QoS选择**：遥测数据用QoS 0，命令用QoS 1
3. **异步处理**：通过Kafka异步处理消息，不阻塞MQTT线程
4. **自动重连**：配置自动重连，保证连接稳定性

**项目中的体现：**
```java
// 连接参数优化
options.setAutomaticReconnect(true);  // 自动重连
options.setKeepAliveInterval(60);     // 心跳间隔

// 异步处理
if (kafkaEnabled) {
    telemetryProducer.sendTelemetry(...);  // 异步发送到Kafka
}
```

### Q8: MQTT在生产环境的安全措施？

**回答要点：**

1. **认证授权**：禁用匿名访问，使用用户名密码认证
2. **传输加密**：启用TLS/SSL加密
3. **网络隔离**：Broker仅监听内网地址
4. **ACL控制**：配置主题访问权限

**项目中的体现：**
```yaml
# mosquitto.conf
allow_anonymous false
password_file /mosquitto/config/passwd
acl_file /mosquitto/config/acl

# docker-compose.yml
ports:
  - "127.0.0.1:1883:1883"  # 仅内网访问
```

DormPower项目使用MQTT作为设备通信协议，主要解决了以下问题：

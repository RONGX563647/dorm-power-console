## 1. Kafka 基础概念

### 1.1 什么是 Kafka

Kafka 是一个分布式流处理平台，由 LinkedIn 开发，现已成为 Apache 开源项目。它具有以下三个核心功能：

1. **消息队列**：发布和订阅消息流
2. **流处理**：实时处理消息流
3. **存储系统**：持久化消息流

### 1.2 Kafka 与其他消息队列对比

| 对比维度 | Kafka | RabbitMQ | ActiveMQ | RocketMQ |
|----------|-------|----------|----------|----------|
| **设计定位** | 分布式流处理 | 传统消息队列 | 传统消息队列 | 分布式消息队列 |
| **性能** | 极高（百万级/秒） | 高（十万级/秒） | 中等 | 高 |
| **持久化** | 磁盘顺序写 | 内存+磁盘 | 磁盘 | 磁盘 |
| **集群支持** | 原生支持 | 需要插件 | 需要插件 | 原生支持 |
| **消息顺序** | 分区内有序 | 支持 | 支持 | 支持 |
| **消息确认** | offset | ack | ack | offset |
| **消费者模式** | pull | push | push | pull |
| **适用场景** | 大数据、日志、流处理 | 企业级应用 | 企业级应用 | 电商、金融 |

## 2. Kafka 核心概念

### 2.1 Topic（主题）

Topic 是消息的分类单位，生产者向特定主题发布消息，消费者订阅特定主题消费消息。

```bash
# 创建主题
kafka-topics.sh --create --topic dorm.telemetry --partitions 3 --replication-factor 1

# 查看主题
kafka-topics.sh --list --bootstrap-server localhost:9092

# 查看主题详情
kafka-topics.sh --describe --topic dorm.telemetry --bootstrap-server localhost:9092
```

### 2.2 Partition（分区）

每个 Topic 可以划分为多个 Partition，实现消息的并行处理和水平扩展。

```
┌─────────────────────────────────────────────────────────────┐
│                      分区机制                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Topic: dorm.telemetry (3个分区)                           │
│                                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐       │
│  │  Partition 0 │  │  Partition 1 │  │  Partition 2 │      │
│  │              │  │              │  │              │      │
│  │  [0] msg1    │  │  [0] msg2    │  │  [0] msg3    │      │
│  │  [1] msg4    │  │  [1] msg5    │  │  [1] msg6    │      │
│  │  [2] msg7    │  │  [2] msg8    │  │  [2] msg9    │      │
│  │              │  │              │  │              │      │
│  │  Broker 1    │  │  Broker 2    │  │  Broker 3    │      │
│  └─────────────┘  └─────────────┘  └─────────────┘       │
│                                                             │
│  消息分配策略：                                             │
│  1. 指定分区：直接发送到指定分区                            │
│  2. 键分区：相同Key发送到相同分区                           │
│  3. 轮询：依次发送到各个分区                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 Replication（副本）

每个 Partition 可以有多个副本，分布在不同的 Broker 上，实现高可用。

```java
// 配置副本因子
Properties props = new Properties();
props.put("replication.factor", 3);
props.put("min.insync.replicas", 2);
```

### 2.4 Consumer Group（消费者组）

消费者组由多个消费者组成，共同消费 Topic 的消息，每个消息只会被组内一个消费者消费。

```
┌─────────────────────────────────────────────────────────────┐
│                    消费者组机制                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Topic: dorm.telemetry (3个分区)                           │
│                                                             │
│  ┌────────────────────────────────────────────────────────┐ │
│  │              Consumer Group: group-1                   │ │
│  │                                                         │ │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐            │ │
│  │  │Consumer 1│  │Consumer 2│  │Consumer 3│            │ │
│  │  │  消费    │  │  消费    │  │  消费    │            │ │
│  │  │ Partition│  │ Partition│  │ Partition│            │ │
│  │  │    0    │  │    1    │  │    2    │            │ │
│  │  └──────────┘  └──────────┘  └──────────┘            │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                             │
│  特点：                                                     │
│  - 每个分区只能被组内一个消费者消费                         │
│  - 增加消费者数量可以提高消费速度                           │
│  - 消费者数量不能超过分区数量                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.5 Offset（偏移量）

Offset 是消息在分区内的唯一标识，用于追踪消费进度。

```java
// 提交偏移量
consumer.commitSync();

// 从特定偏移量开始消费
consumer.seek(topicPartition, offset);

// 获取当前消费位置
consumer.position(topicPartition);

// 提交偏移量到指定位置
consumer.commitSync(Collections.singletonMap(topicPartition, new OffsetAndMetadata(offset)));
```

## 3. Kafka 核心特性

### 3.1 消息顺序性

```
┌─────────────────────────────────────────────────────────────┐
│                      消息顺序性                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  分区内有序：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Partition 0                                         │   │
│  │  [0] create order (orderId=1)                       │   │
│  │  [1] pay order (orderId=1)                          │   │
│  │  [2] ship order (orderId=1)                         │   │
│  │  [3] create order (orderId=2)                       │   │
│  │  [4] pay order (orderId=2)                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  跨分区无序：                                               │
│  ┌─────────────┐  ┌─────────────┐                         │
│  │ Partition 0 │  │ Partition 1 │                         │
│  │  msg1(A)    │  │  msg2(B)    │                         │
│  │  msg3(A)    │  │  msg4(B)    │                         │
│  └─────────────┘  └─────────────┘                         │
│                                                             │
│  保证顺序的方法：                                            │
│  1. 相同业务的消息使用相同 Key                              │
│  2. 发送到一个分区                                          │
│  3. 使用单分区主题                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 消息持久化

Kafka 使用磁盘顺序写实现高性能持久化：

```java
// Producer 配置
props.put("acks", "all");           // 所有副本确认
props.put("retries", 3);             // 重试次数
props.put("batch.size", 16384);     // 批量大小
props.put("linger.ms", 10);         // 等待时间
props.put("buffer.memory", 33554432); // 缓冲区大小
```

### 3.3 消息压缩

Kafka 支持多种压缩算法：

| 压缩算法 | 压缩比 | CPU 使用 | 适用场景 |
|----------|--------|----------|----------|
| gzip | 高 | 高 | 文本、日志 |
| snappy | 中 | 中 | 通用 |
| lz4 | 中 | 低 | 低延迟场景 |
| zstd | 高 | 中 | 最新特性 |

```java
// 配置压缩
props.put("compression.type", "lz4");
```

### 3.4 消息可靠性

```
┌─────────────────────────────────────────────────────────────┐
│                      消息可靠性                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Producer 端：                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  acks=0: 不等待确认（最高性能，最低可靠性）          │   │
│  │  acks=1: Leader确认（平衡）                         │   │
│  │  acks=all: 所有ISR确认（最高可靠性）                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Broker 端：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  replication.factor: 副本因子                       │   │
│  │  min.insync.replicas: 最小同步副本                  │   │
│  │  unclean.leader.election.enable: 选举策略           │   │

│                                                             │
│  Consumer 端：                                             │
│  ┌─────────────────────────────────────────────────────┐   │
```

```
└─────────────────────────────────────────────────────────────┘
```

### 5.3 Leader 选举

```
┌─────────────────────────────────────────────────────────────┐
│                    Leader 选举                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ISR (In-Sync Replicas):                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  与 Leader 保持同步的副本集合                        │   │
│  │  条件:                                              │   │
│  │  - 副本落后 Leader 不超过 replica.lag.time.max.ms   │   │
│  │  - 副本正在同步中                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  选举过程:                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  1. Leader 宕机                                     │   │
│  │  2. Controller 收到通知                            │   │
│  │  3. Controller 选举新的 Leader                     │   │
│  │     - 优先选择 ISR 中的副本                         │   │
│  │     - 其次选择 AR 中的副本                          │   │
│  │  4. 更新元数据                                     │   │
│  │  5. 通知客户端                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  选举策略:                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  unclean.leader.election.enable=false (默认)        │   │
│  │  - 只从 ISR 中选举，确保数据不丢失                  │   │
│  │                                                       │   │
```

## 7. Kafka 面试要点

### 7.1 基础问题

#### Q1: Kafka 为什么快？

**答案**：
1. **顺序写磁盘**：使用磁盘顺序写，接近内存速度
2. **零拷贝**：使用 DMA 直接传输，避免内核用户空间拷贝
3. **批量处理**：消息批量发送和压缩
4. **页缓存**：利用操作系统页缓存
5. **分区分片**：并行处理

#### Q2: Kafka 的 Leader 选举机制？

**答案**：
1. **ISR 集合**：与 Leader 保持同步的副本集合
2. **Controller 选举**：由 ZooKeeper/KRaft 选举产生
3. **选举策略**：优先从 ISR 中选举
4. **客户端感知**：Leader 变化后客户端会感知

#### Q3: Kafka 消息重复消费怎么办？

**答案**：
1. **幂等性**：开启 `enable.idempotence=true`
2. **手动提交**：处理完成后手动提交偏移量
3. **消息去重**：业务层实现去重逻辑
4. **事务**：使用 Kafka 事务

### 7.2 进阶问题

#### Q4: Kafka 如何保证消息顺序？

**答案**：
1. **分区内有序**：同一分区内消息有序
2. **使用 Key**：相同 Key 发送到同一分区
3. **单分区**：使用单分区主题
4. **自定义分区器**：实现特定顺序

#### Q5: Kafka 消费者组再均衡？

**答案**：
1. **触发条件**：消费者加入/离开、分区变化、主题变化
2. **再均衡过程**：停止消费、分配分区、继续消费
3. **影响**：短暂停止服务
4. **优化**：使用静态成员、协作式再均衡

#### Q6: Kafka 消息可靠性如何保证？

**答案**：
1. **Producer 端**：`acks=all` + `retries=3` + 幂等性
2. **Broker 端**：`replication.factor=3` + `min.insync.replicas=2`
3. **Consumer 端**：手动提交偏移量

# 项目H2使用文档

## 一、项目为什么使用H2数据库

### 1.1 技术选型背景

本项目是一个宿舍用电管理系统，生产环境使用PostgreSQL作为主数据库。但在测试和本地开发场景中，我们选择使用H2数据库作为替代方案。

### 1.2 选择H2的核心原因

#### 1.2.1 测试效率提升

```
┌─────────────────────────────────────────────────────────────┐
│                    测试效率对比                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   PostgreSQL测试流程：                                        │
│   启动Docker → 等待数据库就绪 → 执行测试 → 清理数据            │
│   耗时：~30秒/测试套件                                        │
│                                                              │
│   H2测试流程：                                                │
│   启动应用 → 执行测试 → 自动清理                               │
│   耗时：~3秒/测试套件                                         │
│                                                              │
│   效率提升：10倍                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 1.2.2 CI/CD友好

```yaml
# GitHub Actions / Jenkins 无需额外配置数据库服务
# 使用H2后，CI流水线更简洁

# 之前需要：
services:
  postgres:
    image: postgres:15
    env:
      POSTGRES_PASSWORD: postgres
    ports:
      - 5432:5432

# 使用H2后：
# 无需任何外部服务依赖
```

#### 1.2.3 测试隔离性

每个测试类拥有独立的内存数据库实例，测试之间完全隔离：

```java
@SpringBootTest
@ActiveProfiles("test")  // 激活H2配置
class DeviceControllerTest {
    // 独立的H2数据库实例
    // 测试结束后自动销毁
}
```

#### 1.2.4 成本节约

| 环境 | 数据库 | 资源消耗 |
|------|--------|----------|
| 生产环境 | PostgreSQL | 专用服务器 |

| 测试环境 | H2内存模式 | 几乎为零 |
| CI环境 | H2内存模式 | 零额外成本 |

## 二、项目中如何使用H2

### 2.1 Maven依赖配置

```xml
<!-- pom.xml -->
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <version>2.2.224</version>
    <scope>runtime</scope>  <!-- 仅运行时依赖 -->
</dependency>
```

**scope=runtime说明**：
- H2仅在运行时需要，编译时不依赖
- 生产环境部署时不会包含H2

### 2.2 多环境配置策略

项目采用Spring Profile实现多环境配置：

```
backend/src/main/resources/
├── application.yml           # 主配置（PostgreSQL）
├── application-dev.yml       # 开发环境
├── application-local.yml     # 本地环境（H2）
├── application-prod.yml      # 生产环境
└── application-low-memory.yml

backend/src/test/resources/
└── application-test.yml      # 测试环境（H2）
```

### 2.3 测试环境配置详解

```yaml
# application-test.yml
spring:
  datasource:
    # H2内存数据库配置
    url: jdbc:h2:mem:testdb;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE
    username: sa
    password:
    driver-class-name: org.h2.Driver
  
  jpa:
    hibernate:
      ddl-auto: create-drop  # 测试结束自动删除表结构
    show-sql: false
    properties:
      hibernate:
        dialect: org.hibernate.dialect.H2Dialect
  
  h2:
    console:
      enabled: false  # 测试环境禁用控制台
  
  cache:
    type: simple  # 简单缓存，避免Redis依赖

# 禁用外部服务
kafka:
  enabled: false
mqtt:
  enabled: false
```

**配置要点解析**：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `DB_CLOSE_DELAY=-1` | -1 | JVM生命周期内保持数据库打开 |
| `DB_CLOSE_ON_EXIT=FALSE` | FALSE | JVM退出时不自动关闭 |
| `ddl-auto=create-drop` | create-drop | 启动创建表，关闭删除表 |
| `dialect=H2Dialect` | H2Dialect | 使用H2方言 |

### 2.4 本地开发环境配置

### 2.5 测试类使用示例

#### 2.5.1 集成测试示例

```java
 * 认证控制器集成测试
 * 使用H2内存数据库进行测试
 */

    @Test

        String loginJson = """
            {
                "account": "admin",
                "password": "admin123"
            }
            """;

        mockMvc.perform(post("/api/auth/login")
                .contentType(MediaType.APPLICATION_JSON)
                .content(loginJson))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.token").exists());
    }
}
```

## 三、面试高频问题与回答

### 3.1 项目中为什么使用H2数据库？

**标准回答**：

> 在我们的宿舍用电管理系统中，生产环境使用PostgreSQL作为主数据库，但在测试和本地开发环境选择H2数据库，主要基于以下考虑：
>
> **第一，测试效率**：H2是纯内存数据库，启动速度极快，测试执行时间从30秒降低到3秒，效率提升10倍。
>
> **第二，测试隔离**：每个测试类拥有独立的内存数据库实例，测试之间完全隔离，避免了测试数据污染。
>
> **第三，CI/CD友好**：无需在CI流水线中配置外部数据库服务，简化了部署流程，降低了CI资源消耗。
>
> **第四，成本节约**：开发和测试环境无需额外的数据库服务器资源。
>
> 我们通过Spring Profile机制实现多环境配置切换，测试环境自动激活H2配置。

### 3.2 H2数据库和PostgreSQL有什么区别？如何保证兼容性？

**标准回答**：

> H2和PostgreSQL的主要区别在于：
>
> **存储方式**：H2支持内存模式和文件模式，PostgreSQL是磁盘存储的客户端-服务器架构。
>
> **并发能力**：PostgreSQL支持高并发，H2并发能力有限。
>
> **企业特性**：PostgreSQL有完善的备份恢复、复制、分区等企业级特性。
>
> **兼容性保证**：
> 1. 我们使用H2的PostgreSQL兼容模式：`MODE=PostgreSQL`
> 2. JPA层使用统一的Repository接口，对上层透明
> 3. 核心业务逻辑测试在H2上执行，关键集成测试在真实PostgreSQL上验证
> 4. 避免使用数据库特有函数，使用JPA标准API

### 3.3 H2的DB_CLOSE_DELAY参数有什么作用？

**标准回答**：

> `DB_CLOSE_DELAY=-1`是H2数据库的关键配置参数，它的作用是让内存数据库在JVM生命周期内保持打开状态。
>
> 默认情况下，H2内存数据库在最后一个连接关闭时会自动删除数据。但在测试场景中，我们需要在多个测试方法之间共享同一个数据库实例，如果不设置这个参数，每个测试方法执行完后数据库就会被清空。
>
> 设置为-1表示数据库会一直保持打开，直到JVM退出。这样保证了测试过程中数据库状态的持续性。

### 3.4 如何在项目中实现多环境数据库切换？

**标准回答**：

> 我们使用Spring Profile机制实现多环境数据库切换：
>
> **配置层面**：
> - `application.yml`：主配置，使用PostgreSQL
> - `application-test.yml`：测试配置，使用H2
> - `application-local.yml`：本地开发配置，使用H2
>
> **激活方式**：
> - 测试类使用`@ActiveProfiles("test")`注解
> - 启动参数`--spring.profiles.active=local`
>
> **代码层面**：
> Repository层使用Spring Data JPA，对底层数据库透明，无需修改代码。

```java
// 测试类示例
@SpringBootTest
@ActiveProfiles("test")  // 自动使用H2配置
class MyTest {
    // ...
}
```

### 3.5 H2数据库适合用在生产环境吗？为什么？

**标准回答**：

> H2数据库一般不建议用于高并发的生产环境，原因如下：
>
> **数据安全**：内存模式下数据易丢失，文件模式缺乏完善的事务日志和恢复机制。
>
> **并发能力**：H2的并发处理能力有限，无法应对高并发场景。
>
> **企业特性**：缺乏PostgreSQL/MySQL等数据库的成熟特性，如主从复制、读写分离、分区表等。
>
> **适用场景**：
> - 单元测试和集成测试
> - 原型开发和POC验证
> - 嵌入式应用（如桌面应用内置数据库）
> - CI/CD流水线
>
> 在我们项目中，H2仅用于测试和本地开发环境，生产环境使用PostgreSQL。

### 3.6 你们的测试策略是怎样的？H2在其中扮演什么角色？

**标准回答**：

> 我们的测试策略采用分层测试金字塔：
>
> **单元测试**：使用Mockito模拟依赖，不涉及数据库，速度最快。
>
> **集成测试**：使用H2内存数据库，测试Repository层和Service层的协作，保证测试隔离和速度。
>
> **端到端测试**：在真实PostgreSQL环境执行，验证完整的业务流程。
>
> H2在集成测试中扮演核心角色：
> 1. 提供快速的数据库环境
> 2. 支持create-drop模式，自动管理表结构
> 3. 与JPA完美集成，测试代码无需修改

```
        ┌─────────────┐
        │  E2E Test   │  ← PostgreSQL
        │   (少量)     │
        ├─────────────┤
        │ Integration │  ← H2 Database
        │   Test      │
        ├─────────────┤
        │   Unit      │  ← Mockito
        │   Test      │
        └─────────────┘
```

## 四、项目配置速查表

### 4.1 环境配置对照

| 环境 | Profile | 数据库 | ddl-auto | 外部服务 |
|------|---------|--------|----------|----------|
| 生产 | prod | PostgreSQL | update | 全部启用 |
| 开发 | dev | PostgreSQL | create-drop | 部分启用 |
| 本地 | local | H2内存 | create-drop | 全部禁用 |
| 测试 | test | H2内存 | create-drop | 全部禁用 |

### 4.2 H2配置参数速查

| 参数 | 值 | 说明 |
|------|-----|------|
| `DB_CLOSE_DELAY` | -1 | 保持数据库打开 |
| `DB_CLOSE_ON_EXIT` | FALSE | JVM退出不关闭 |
| `MODE` | PostgreSQL | 兼容模式 |
| `AUTO_SERVER` | TRUE | 多进程访问 |

### 4.3 常用命令

```bash

# 本地开发启动（使用H2）

# 访问H2控制台（本地开发时）
http://localhost:8000/h2-console
JDBC URL: jdbc:h2:mem:testdb
User: sa
Password: (空)
```

## 五、总结

H2数据库在本项目中的定位非常清晰：**测试和开发环境的轻量级数据库解决方案**。它帮助我们：

1. **提升测试效率**：测试执行速度提升10倍
2. **简化开发流程**：本地开发无需配置外部数据库
3. **优化CI/CD**：流水线无需数据库服务依赖
4. **降低资源成本**：节省开发和测试环境的服务器资源

通过Spring Profile机制，我们实现了多环境的无缝切换，代码层面对底层数据库完全透明，体现了良好的架构设计。

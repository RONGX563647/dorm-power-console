## 二、需求分析与设计

### 2.1 业务场景

**场景 1：设备数据上报**
```
智能插座 → MQTT Broker → MqttBridge → 数据验证 → 存储到 Telemetry 表 → WebSocket 推送 → 前端实时更新
```

**场景 2：历史数据查询**
```
用户选择设备和时间范围 → 后端查询 Telemetry 表 → 分页返回 → 前端渲染图表
```

**场景 3：实时监控**
```
设备上报数据 → WebSocket 推送 → 前端订阅 → 实时更新图表（1 秒刷新）
```

**场景 4：数据导出**
```
用户选择导出范围 → 后端生成 CSV 文件 → 返回下载链接 → 用户下载
```

### 2.2 功能需求清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 遥测数据接收 | 通过 MQTT 接收设备上报数据 | P0 |

| 实时数据推送 | WebSocket 推送实时数据 | P0 |

| 系统指标采集 | CPU、内存、设备在线率 | P2 |
| 定时任务管理 | 数据清理等定时任务 | P2 |

### 2.3 技术选型与理由

| 技术 | 说明 | 版本 | 选型理由 |
|------|------|------|----------|
| **MQTT** | 设备数据上报协议 | 3.1.1 | ✅ **发布/订阅模式**：设备发布数据，服务端订阅消费<br>✅ **主题过滤**：支持通配符主题（dorm/telemetry/#）<br>✅ **QoS 保障**：关键数据使用 QoS 1 确保到达<br>✅ **低延迟**：适合高频实时数据传输 |
| **WebSocket** | 实时数据推送 | - | ✅ **全双工通信**：服务端主动推送数据到前端<br>✅ **低延迟**：一次握手，持续通信<br>✅ **轻量协议**：相比 HTTP 轮询，减少带宽消耗<br>✅ **Spring 支持**：Spring WebSocket 简化开发 |
| **Spring Scheduler** | 定时任务调度 | 3.2.x | ✅ **内置支持**：Spring Boot 内置，无需额外依赖<br>✅ **Cron 表达式**：灵活的调度策略<br>✅ **异步执行**：支持@Async 异步任务<br>⚠️ **局限性**：单机调度，分布式场景需用 Quartz/XXL-JOB |
| **ECharts** | 图表库 | 5.x | ✅ **丰富图表**：支持折线图、柱状图、仪表盘等<br>✅ **大数据量**：支持 canvas 渲染，性能优秀<br>✅ **响应式**：自适应不同屏幕尺寸<br>✅ **中文文档**：百度开源，文档完善 |

**技术对比与取舍**：
- **WebSocket vs SSE**：选择 WebSocket 因为需要双向通信（前端也可发送控制命令）
- **ECharts vs Chart.js**：选择 ECharts 因为功能更丰富，适合复杂的数据可视化场景
- **定时任务 vs 实时计算**：教学场景用定时任务简化实现，生产环境考虑 Flink 实时计算
| PostgreSQL | 时序数据存储 | 14+ |

```java
package com.dormpower.model;

import jakarta.persistence.*;
import lombok.Data;
import java.math.BigDecimal;
import java.time.LocalDateTime;
import java.util.UUID;

@Data
@Entity
@Table(name = "telemetry", indexes = {
    @Index(name = "idx_telemetry_device_time", columnList = "deviceId, timestamp"),
    @Index(name = "idx_telemetry_timestamp", columnList = "timestamp")
})
public class Telemetry {
    
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;
    
    @Column(name = "device_id", nullable = false)
    private UUID deviceId;
    
    @Column(precision = 10, scale = 2)
    private BigDecimal voltage; // 电压（V）
    
    @Column(precision = 10, scale = 3)
    private BigDecimal current; // 电流（A）
    
    @Column(precision = 10, scale = 2)
    private BigDecimal power; // 功率（W）
    
    @Column(precision = 5, scale = 3)
    private BigDecimal powerFactor; // 功率因数
    
    @Column(precision = 10, scale = 2)
    private BigDecimal frequency; // 频率（Hz）
    
    @Column(precision = 12, scale = 4)
    private BigDecimal activeEnergy; // 有功电能（kWh）
    
    @Column(precision = 12, scale = 4)
    private BigDecimal reactiveEnergy; // 无功电能（kVarh）
    
    @Column(precision = 5, scale = 2)
    private BigDecimal temperature; // 温度（℃）
    
    @Column(precision = 5, scale = 2)
    private BigDecimal humidity; // 湿度（%）
    
    @Column(nullable = false)
    private LocalDateTime timestamp; // 数据采集时间
    
    private LocalDateTime createdAt = LocalDateTime.now();
    
    private Integer quality; // 数据质量标识
}
```

**DTO 设计原则**:

```
┌─────────────────────────────────────────────┐
│            DTO 设计最佳实践                   │
├─────────────────────────────────────────────┤
│ 1. 职责单一:                                 │
│    - Request DTO: 接收前端数据,添加验证       │
│    - Response DTO: 返回前端数据,过滤敏感信息  │
│    - Message DTO: 消息传递载体(MQTT/WebSocket)│
│                                            │
│ 2. 命名规范:                                 │
│    - 请求DTO: XxxRequest.java                │
│    - 响应DTO: XxxResponse.java               │
│    - 消息DTO: XxxMessage.java                │
│                                            │
│ 3. 验证注解:                                 │
│    - @NotNull: 不能为null                    │
│    - @NotBlank: 不能为空白字符串              │
│    - @DecimalMin/DecimalMax: 数值范围验证     │
│    - @Pattern: 正则表达式验证                 │
│                                            │
│ 4. API文档:                                  │
│    - @Schema: OpenAPI 3.0 文档注解           │
│    - description: 字段说明                   │
│    - example: 示例值                         │
│    - required: 是否必填                      │
│                                            │
│ 5. 与Entity的区别:                           │
│    - Entity: 数据库映射,包含JPA注解           │
│    - DTO: 数据传输,包含验证注解               │
│    - 两者可以相互转换,使用Mapper或手动转换    │
│                                            │
│ 6. Builder模式:                              │
│    - 响应DTO推荐使用Builder模式               │
│    - 链式调用,代码更简洁                      │
│    - 避免构造函数参数过多                     │
└─────────────────────────────────────────────┘
```

### 6.2 常见问题与解决方案

#### 问题 1：大量数据写入导致数据库压力大
**现象**: 数据写入慢，CPU 占用高

**解决方案**:
```java
// 使用批量插入
@Transactional
public void batchSaveTelemetry(List<TelemetryData> dataList) {
    List<Telemetry> telemetryList = dataList.stream()
        .map(data -> modelMapper.map(data, Telemetry.class))
        .collect(Collectors.toList());
    
    // 分批保存，每 1000 条一批
    int batchSize = 1000;
    for (int i = 0; i < telemetryList.size(); i += batchSize) {
        int end = Math.min(i + batchSize, telemetryList.size());
        telemetryRepository.saveAll(telemetryList.subList(i, end));
    }
}
```

#### 问题 2：实时数据推送延迟
**现象**: 前端收到数据延迟超过 1 秒

**解决方案**:
```java
// 使用异步推送
@Async
public void sendTelemetryUpdate(Telemetry telemetry) {
    String message = objectMapper.writeValueAsString(telemetry);
    webSocketManager.sendToTopic("telemetry:" + telemetry.getDeviceId(), message);
}
```

#### 问题 3：图表数据点过多渲染慢
**现象**: 图表卡顿，浏览器 CPU 占用高

**解决方案**:
```typescript
// 数据降采样
function downsample(data: number[], threshold: number) {
    if (data.length <= threshold) return data;
    
    const sampled: number[] = [];
    const step = Math.floor(data.length / threshold);
    
    for (let i = 0; i < data.length; i += step) {
        sampled.push(data[i]);
    }
    
    return sampled;
}
```

#### 2.2.1 优先级评估

```java
// 技术债务优先级计算公式
public class DebtPrioritizer {
    
    public double calculatePriority(DebtItem item) {
        // 优先级 = (影响范围 * 严重程度) / 修复成本
        double impact = calculateImpact(item);      // 1-10
        double severity = calculateSeverity(item);  // 1-10
        double cost = calculateCost(item);          // 1-10
        
        return (impact * severity) / cost;
    }
    
    private double calculateImpact(DebtItem item) {
        // 影响的代码行数
        // 影响的接口数量
        // 影响的模块数量
        return item.getAffectedLines() / 100.0;
    }
    
    private double calculateSeverity(DebtItem item) {
        switch (item.getType()) {
            case SECURITY: return 10.0;    // 安全问题最高优先级
            case PERFORMANCE: return 8.0;  // 性能问题高优先级
            case MAINTAINABILITY: return 6.0; // 可维护性中等
            case STYLE: return 3.0;        // 代码风格低优先级
            default: return 5.0;
        }
    }
    
    private double calculateCost(DebtItem item) {
        // 预估修复时间（小时）
        // 需要修改的文件数
        // 需要的测试工作量
        return item.getEstimatedHours();
    }
}
```

```java
// 阶段1: 识别问题
// SonarQube报告：Performance issue - N+1 query detected
// 代码位置：DeviceService.getAllDevicesWithStatus()

// 阶段2: 分析问题
// 问题：查询所有设备后，循环查询每个设备的状态
// 影响：1000个设备 = 1 + 1000 = 1001次数据库查询
// 耗时：1001 * 5ms = 5005ms ≈ 5秒

// 阶段3: 制定修复方案
// 方案A: 使用JOIN FETCH（适合关联数据）
// 方案B: 批量查询（适合独立查询）
// 方案C: 缓存（适合不频繁变化的数据）

// 阶段4: 实施方案A
@Repository
public interface DeviceRepository extends JpaRepository<Device, String> {
    
    // 修复前
    @Query("SELECT d FROM Device d")
    List<Device> findAll();
    
    // 修复后
    @Query("SELECT DISTINCT d FROM Device d LEFT JOIN FETCH d.status")
    List<Device> findAllWithStatus();
}

// 阶段5: 验证效果
// 查询次数：1001次 -> 1次
// 响应时间：5005ms -> 50ms
// 性能提升：100倍
```

```python
# 边缘设备上的轻量级AI推理
class EdgeAIModel:
    """在边缘网关运行的轻量级模型"""
    
    def __init__(self, model_path: str):
        # 使用TensorFlow Lite或ONNX Runtime
        self.interpreter = tflite.Interpreter(model_path=model_path)
        self.interpreter.allocate_tensors()
    
    def predict(self, sensor_data: np.ndarray) -> dict:
        """本地推理，无需云端"""
        # 数据预处理
        input_data = self.preprocess(sensor_data)
        
        # 推理
        self.interpreter.set_tensor(0, input_data)
        self.interpreter.invoke()
        
        # 后处理
        output = self.interpreter.get_tensor(0)
        return self.postprocess(output)
    
    def preprocess(self, data: np.ndarray) -> np.ndarray:
        # 标准化、降采样等
        return (data - self.mean) / self.std
    
    def postprocess(self, output: np.ndarray) -> dict:
        # 阈值判断、告警生成
        return {
            "anomaly_score": float(output[0]),
            "should_alert": output[0] > self.threshold
        }
```

```python
class PredictiveMaintenance:

        self.model = self.load_forecasting_model()
    
    def predict_failure_risk(self, device_id: str, horizon: int = 7) -> dict:

        if risk_score > 0.8:
            action = "立即检修"
        elif risk_score > 0.6:
            action = "计划检修"
        elif risk_score > 0.4:
            action = "加强监控"

            action = "正常运行"

            "risk_score": risk_score,
            "prediction_horizon": f"{horizon}天",
```

```yaml
      - 边界条件处理
      - 异常处理完善
    
    2-性能:
      - 有无性能问题
      - 数据库查询优化
      - 内存使用合理
    
    3-可维护性:
      - 代码结构清晰
      - 命名规范
      - 注释充分
    
    4-安全性:
      - SQL注入防护
      - XSS防护
      - 权限校验
    
    5-测试:
      - 单元测试覆盖
      - 测试用例完整
      - 边界测试
  
  审查流程:
    1. 提交Pull Request
    2. CI自动检查通过
    3. 至少1人Review通过
    4. 解决所有Review意见
    5. 合并到主分支
```

### 5.3 故障复盘机制

```yaml
Post-mortem流程:
  触发条件:
    - P0级故障（系统不可用）
    - P1级故障（核心功能受损）
    - 数据丢失或损坏
    - 安全事件
  
  复盘时间:
    - 故障解决后48小时内
    - 避免指责文化（Blameless）

      - 故障发生时间

    4-经验总结:
```

## 七、实战案例

### 7.1 案例一：从单体到微服务的演进

```yaml
背景:
  - 用户量从1000增长到50000
  - 数据库连接数成为瓶颈
  - 部署时间从5分钟增长到30分钟
  - 团队规模从5人增长到30人

问题:
  - 代码冲突频繁
  - 构建时间长（20分钟）
  - 部署风险高（牵一发动全身）
  - 扩展性差（无法按模块扩展）

演进步骤:
  阶段1-模块解耦（2个月）:
    - 按业务划分模块边界
    - 定义模块间API
    - 移除循环依赖
    - 建立模块测试
  
  阶段2-数据拆分（1个月）:
    - 识别数据归属
    - 拆分数据库Schema
    - 建立数据同步机制
    - 灰度迁移数据
  
  阶段3-服务独立部署（2个月）:
    - 拆分CI/CD流水线
    - 独立数据库实例
    - 建立服务注册发现
    - 引入API网关
  
  阶段4-服务治理（1个月）:
    - 限流降级
    - 熔断机制
    - 链路追踪
    - 监控告警

成果:
  - 部署时间：30分钟 -> 5分钟
  - 构建时间：20分钟 -> 3分钟
  - 可用性：99% -> 99.9%
  - 团队效率：提升3倍
```

### 7.2 案例二：性能优化实战

```yaml
背景:
  - 设备数据采集延迟>5秒
  - API接口响应时间>3秒
  - 数据库CPU持续90%+
  - 用户投诉系统卡顿

问题定位:
  1-慢查询分析:
    - 发现30+个慢查询（>1秒）
    - 缺少关键索引
    - N+1查询问题严重
  
  2-内存分析:
    - 发现内存泄漏（静态缓存）
    - Full GC频繁（每天10+次）
    - 对象创建速率过高
  
  3-网络分析:
    - MQTT连接频繁断开重连
    - 大批次数据传输
    - 缺少连接池

优化措施:
  数据库:
    - 创建15个关键索引
    - 优化20个慢查询
    - 引入读写分离
    - 批量插入改为500条/批
  
  缓存:
    - 引入Caffeine本地缓存
    - 热点数据Redis缓存
    - 缓存穿透保护（布隆过滤器）
    - 缓存更新策略优化
  
  代码:
    - 修复N+1查询（15处）
    - 引入虚拟线程池
    - 优化对象创建（对象池）
    - 异步处理非关键路径

优化结果:
  - 数据采集延迟：5秒 -> 0.5秒（10倍）
  - API响应时间：3秒 -> 0.2秒（15倍）
  - 数据库CPU：90% -> 40%
  - Full GC：每天10+次 -> 每周1次
```
